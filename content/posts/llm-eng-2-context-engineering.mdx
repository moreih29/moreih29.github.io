1편에서 프롬프트 엔지니어링의 한계를 정리하며 이런 질문을 남겼다. 모델이 모르는 것을 프롬프트로 어떻게 해결하는가? 긴 문서를 어떻게 효과적으로 활용하는가? 세션 간 기억은 어떻게 관리하는가?

이 질문들의 공통점이 있다. 모두 **무엇을 모델에게 보여줄지**의 문제다. 프롬프트 문장을 어떻게 쓰는가가 아니라, 컨텍스트 창 전체를 어떻게 채울지의 문제다.

이것이 컨텍스트 엔지니어링이다.

---

## 컨텍스트 엔지니어링이란

2025년 6월 Andrej Karpathy가 X(구 트위터)에서 "context engineering"이라는 용어를 지지하면서 이 개념이 빠르게 확산됐다 [T3]. Shopify CEO Tobi Lutke가 동조했고, 이후 Anthropic이 엔지니어링 블로그에서 공식 채택했다 [T2].

Karpathy의 정의는 간결하다. "컨텍스트 엔지니어링은 다음 스텝을 위해 컨텍스트 창을 적절한 정보로 채우는 섬세한 예술이자 과학이다."

비유도 명확하다. LLM이 CPU라면 컨텍스트 창은 RAM이고, 엔지니어는 OS다. OS가 어떤 데이터를 RAM에 올릴지 결정하듯, 엔지니어는 어떤 정보를 컨텍스트 창에 넣을지 설계한다.

Anthropic은 이를 이렇게 정리한다 [T2]: "언어 모델로 빌드하는 것은 프롬프트에서 올바른 단어와 구절을 찾는 것이 덜 중요해지고, '모델의 원하는 행동을 가장 잘 생성할 컨텍스트 구성이 무엇인가'라는 더 넓은 질문에 답하는 것이 더 중요해지고 있다."

컨텍스트 창을 채우는 구성 요소들:

- **태스크 설명**: 시스템 프롬프트, 지시 (L1에서 다뤘다)
- **Few-shot 예시**: 패턴 제시
- **RAG 검색 결과**: 외부 지식
- **도구 출력**: Function Calling 결과
- **대화 히스토리**: 이전 턴
- **메모리**: 세션 간 지속 정보
- **압축된 컨텍스트**: 요약, 필터링된 정보

이 구성 요소들을 효율적으로 관리하는 것이 2편의 주제다.

## RAG의 4단계 진화

RAG(Retrieval-Augmented Generation, 검색 증강 생성)는 컨텍스트 엔지니어링의 가장 성숙한 기법이다. 2020년 기본 형태에서 2025년 에이전틱 형태까지 4단계로 진화했다.

<figure>
  <img src="/images/llm-eng/rag-evolution.svg" alt="RAG 4단계 진화 다이어그램 — Naive RAG, Advanced RAG, Modular RAG, Agentic RAG" />
  <figcaption>RAG 4단계 진화. 단순 벡터 검색에서 에이전트가 검색 과정을 동적으로 제어하는 형태로 발전했다.</figcaption>
</figure>

RAG의 기본 파이프라인(임베딩, 벡터 검색, 생성)은 "LLM 해부학 4편"에서 다루므로 여기서는 생략한다. 각 단계의 **실무적 전환점**에 집중한다.

### 1단계: Naive RAG (2020~2022)

Lewis et al.(2020)이 제안한 기본 구조다. 문서를 청크로 분할하고, 임베딩해서 벡터 DB에 저장한다. 쿼리가 들어오면 유사도 기반 Top-K를 검색해 LLM에 넘긴다.

문제는 금방 드러났다. 세 가지가 핵심이다.

**낮은 정밀도/재현율**: 단순 코사인 유사도는 어휘 불일치에 취약하다. "심근경색"을 "심장마비"로 물으면 놓친다.

**청크 경계 문맥 손실**: 문서를 자르면 맥락이 끊긴다. 청크 하나만 보면 무슨 주제인지 모를 수 있다.

**단일 쿼리 한계**: "A와 B의 차이를 설명하고, 각각 언제 쓰는지 알려줘" 같은 복합 질문은 단일 검색으로 처리하기 어렵다.

### 2단계: Advanced RAG (2023~2024)

Gao et al.(2023) 서베이가 이 단계를 체계화했다. 핵심은 검색 전후에 최적화 단계를 추가한다는 것이다.

**Pre-retrieval (쿼리 최적화)**:
- **Query Rewriting**: 사용자 질문을 검색 최적화 형태로 재작성
- **HyDE(Hypothetical Document Embeddings)**: 쿼리로 가상의 답변 문서를 생성하고, 그 문서의 임베딩으로 검색. 쿼리보다 실제 문서와 더 가까운 임베딩 공간에서 검색
- **Query Expansion**: 쿼리를 여러 변형으로 확장해 병렬 검색

**Post-retrieval (결과 정제)**:
- **Hybrid Search**: BM25(키워드 기반)와 Dense Vector(의미 기반)를 RRF(Reciprocal Rank Fusion)로 결합. 둘 중 하나가 놓친 것을 다른 쪽이 잡는다
- **Reranking**: Cross-encoder로 검색 결과를 다시 정렬. bi-encoder 검색보다 정밀도가 높지만 느림. 상위 20개를 검색하고 상위 5개로 재순위화하는 패턴이 일반적

**Anthropic Contextual Retrieval (2024.09)** [T2]은 청크 문맥 손실 문제를 직접 해결했다. 각 청크를 임베딩하기 전에 문서 전체 맥락 요약을 prepend한다. "이 청크가 전체 문서에서 어떤 맥락인지"를 임베딩에 반영하는 것이다.

효과가 뚜렷했다. Top-20 검색 실패율이 35% 감소했다(5.7% → 3.7%). Contextual Embeddings에 BM25를 결합하면 49% 감소(5.7% → 2.9%), Reranking까지 추가하면 67% 개선(5.7% → 1.9%)이 됐다.

### 3단계: Modular RAG (2024)

파이프라인의 각 단계를 교체 가능한 모듈로 분리하는 설계다. Retriever, Generator, Evaluator, Router가 독립 모듈로 동작하고, 에이전트가 이를 오케스트레이션한다.

"Modular RAG"의 정의는 구현마다 다르다. 공통 핵심은 하나다. **파이프라인의 모듈화와 교체 가능성**. 검색기를 바꾸고 싶으면 그 모듈만 교체하면 된다. 특정 태스크에 최적화된 검색 전략을 동적으로 선택할 수 있다.

### 4단계: Agentic RAG (2025~)

Sing et al.(2025) 서베이가 이 단계를 정의한다. 에이전트가 검색 과정 자체를 동적으로 제어한다.

핵심 변화는 네 가지다. **동적 의사결정** — 검색이 필요한지 판단한다. **반복적 검색** — 첫 검색 결과가 불충분하면 다시 검색한다. **자기 반성** — 검색 결과와 생성 결과의 품질을 스스로 평가한다. **멀티 에이전트 협업** — 복잡한 쿼리를 여러 에이전트가 분담한다.

최신 방향은 RL 기반이다. Search-O1, Search-R1은 "언제 검색할지, 무엇을 검색할지"를 강화학습으로 학습한다. MCTS-RAG는 Monte Carlo Tree Search로 검색 경로를 탐색한다.

Agentic RAG의 에이전트 패턴 자체는 "LLM 해부학 8편"에서 다룬다. 여기서는 RAG와의 결합이 실무에서 무엇을 바꾸는지에 집중한다. 가장 큰 변화는 "검색을 할지 말지"를 하드코딩에서 모델 판단으로 옮긴다는 것이다.

**프로덕션 사례:**

Morgan Stanley의 AskResearchGPT [T2]는 7만 건 이상의 리서치 보고서를 Agentic RAG로 서빙한다. 640억 달러 순자산 달성에 기여했고 Celent 2024 Award를 수상했다.

Salesforce Agentforce [T2]는 멀티소스 Agentic RAG로 Fisher & Paykel의 외부 문의 66%, 내부 문의 84%를 자동 처리한다. 월 20억 건의 예측을 처리하고 있다.

Klarna AI [T2]는 2024년 1개월 동안 230만 건을 처리했다. 해결 시간 80% 단축, 비용 4000만 달러 절감이다. 다만 이 사례는 AI 도입으로 인한 직원 감원 논란을 함께 언급할 필요가 있다. 효율성 수치만으로 평가하기 어려운 맥락이 있다.

## 메모리 시스템

RAG가 "지식"의 문제를 다룬다면, 메모리 시스템은 "기억"의 문제를 다룬다. LLM은 기본적으로 상태가 없다. 세션이 끝나면 대화 내용을 기억하지 못한다.

### 메모리 세 계층

**In-context (단기 메모리)**: 현재 컨텍스트 창 안의 모든 것. 시스템 프롬프트, 대화 히스토리, 도구 정의가 여기 속한다. 빠르고 즉각적이지만 컨텍스트 창이 한계다.

**External (장기 메모리)**: 벡터 DB, 키-값 스토어. 세션 간 지속된다. RAG로 필요한 것만 검색해 컨텍스트에 주입한다. 용량 제한이 없지만 검색 지연과 인프라가 필요하다.

**In-weights (내재화 메모리)**: 파인튜닝으로 모델 파라미터에 저장된 지식. 가장 빠르지만 업데이트 비용이 높고 정확도가 낮을 수 있다.

장기 메모리의 종류도 세 가지로 구분한다. **Semantic** — "무엇"을 알고 있는가. 사실, 개념. **Episodic** — "언제/어디서" 무슨 일이 있었는가. 과거 대화, 상호작용 기록. **Procedural** — "어떻게" 하는가. 규칙, 선호도, 절차.

### 대화 히스토리 관리

컨텍스트 창이 길어질수록 비용과 지연이 증가한다. 오래된 대화를 어떻게 처리할지는 실무의 핵심 결정이다.

**Sliding Window**: 최근 N개 메시지만 유지한다. 구현이 단순하지만 오래된 맥락을 완전히 잃는다.

**Recursive Summarization** (Lee et al., 2023): LLM으로 오래된 대화 구간을 요약하고 요약본을 컨텍스트에 포함한다. 정보 손실이 적지만 요약 자체가 LLM 호출이므로 추가 비용이 발생한다.

**Assertion Database**: 대화를 사실 주장(assertion) DB로 변환하고, 관련 사실만 RAG로 검색해 컨텍스트에 주입한다. 가장 정밀하지만 구현이 복잡하다.

### 패턴 선택 가이드

| 상황 | 권장 패턴 |
|------|-----------|
| 단일 세션, 짧은 대화 | In-context window (추가 인프라 없음) |
| 단일 세션, 긴 대화 | Sliding Window + Recursive Summarization |
| 멀티 세션, 장기 선호 저장 | External Vector Store (Semantic Memory) |
| 절차적 규칙 누적 | Procedural Memory / In-weights |
| 장기 에이전틱 태스크 | Episodic + Archival Memory |

### MemGPT vs Mem0

장기 메모리 관리의 두 접근법이 실무에서 비교된다.

**MemGPT/Letta** [T1]는 LLM을 OS처럼 다룬다. 에이전트가 function call로 Core Memory(RAM 역할)와 Archival/Recall Memory(디스크 역할) 사이에서 직접 정보를 이동시킨다. 에이전트가 자율적으로 "무엇을 기억할지"를 결정한다. 투명하고 연구 친화적이지만 런타임 복잡도가 높고 추가 오버헤드가 있다.

**Mem0** [T2]는 메모리를 서비스로 제공하는 경량 레이어다. Extraction Phase에서 대화의 중요 정보를 추출하고, Update Phase에서 기존 메모리와 비교해 갱신/삭제한다. 어떤 프레임워크에도 독립적으로 붙일 수 있다. p95 지연 91% 감소, 토큰 비용 90% 이상 절감을 보고한다.

요약하면: MemGPT는 에이전트가 기억을 직접 운영하고, Mem0는 메모리를 외부 서비스로 제공한다. 에이전트의 자율성이 중요하면 MemGPT, 기존 시스템에 메모리를 빠르게 붙이고 싶으면 Mem0가 적합하다.

## 컨텍스트 창 관리

컨텍스트 창이 길어질수록 두 가지 문제가 생긴다. 비용과 품질이다.

### Lost in the Middle

Liu et al.(2023)은 "Lost in the Middle" 현상을 실증했다. 관련 정보가 긴 컨텍스트의 **중간에 위치하면 성능이 크게 저하**된다. 시작과 끝에 위치할 때 모델이 가장 잘 활용한다.

실용 해결책은 단순하다. 가장 중요한 정보를 프롬프트 시작이나 끝에 배치한다. RAG 검색 결과를 넣을 때도 가장 관련성 높은 청크를 앞뒤에 놓는다. 이를 **Position Engineering**이라 부르기도 한다.

### Context Stuffing vs RAG 선택 기준

컨텍스트 창이 충분히 크다면(1M 토큰 등) 모든 문서를 그냥 넣는 것이 간단하다. 이를 Context Stuffing이라 한다. 하지만 항상 좋은 선택은 아니다.

Context Stuffing이 불리한 경우:
- 1M 컨텍스트는 처리에 약 60초, 비용 $0.50~$20 수준
- "Lost in the Middle"로 품질 저하 가능
- 빈번히 업데이트되는 정보라면 매번 전체 컨텍스트를 재구성해야 함

RAG가 유리한 경우:
- 검색 스타일 쿼리 (특정 정보를 찾는 작업)
- 자주 업데이트되는 데이터
- 비용에 민감한 환경

Long Context가 유리한 경우:
- 지연에 덜 민감한 배치 처리
- 문서 전체를 함께 봐야 하는 작업 (비교 분석, 전체 맥락 이해)

### Prompt Caching

반복적인 프롬프트 prefix를 캐싱하는 기법이다.

Anthropic이 2024년 12월 GA로 출시한 Prompt Caching [T2]은 시스템 프롬프트나 긴 참조 문서처럼 반복되는 prefix를 서버 측에서 캐싱한다. 기본 TTL 5분, 1시간 옵션이 있다. 비용 41~80% 절감, 첫 토큰 생성 시간(TTFT) 13~31% 개선을 보고한다.

실무 전략: 시스템 프롬프트와 정적 참조 문서를 캐시 대상으로 설정하고, 사용자 입력이나 동적 도구 결과는 캐시 밖에 둔다.

주의점이 있다. "Don't Break the Cache" 논문(arXiv:2601.06007) [T1]은 장기 에이전틱 태스크에서 전체 컨텍스트를 무조건 캐시하면 오히려 지연이 증가할 수 있다고 경고한다. 캐시가 유효한 구간과 동적으로 변하는 구간을 구분하는 것이 중요하다.

### 컨텍스트 압축

컨텍스트가 너무 길다면 압축도 선택지다.

**LLMLingua 시리즈** (Microsoft Research): encoder-only LM으로 토큰 수준 필터링을 한다. 중요도가 낮은 토큰을 제거해 컨텍스트를 압축한다. LLMLingua-2는 속도와 압축률을 개선했다.

**ACC-RAG**: 쿼리 복잡도에 따라 압축률을 동적으로 조절한다. 단순 쿼리에는 강하게 압축하고 복잡한 쿼리에는 덜 압축한다. 표준 RAG 대비 4배 빠른 추론을 보고한다.

**AttentionRAG (2025)**: Attention 점수 기반 pruning. 모델이 실제로 주목하는 부분을 기준으로 나머지를 제거한다.

압축은 항상 품질 트레이드오프가 있다. 정확도가 중요한 작업에서는 압축보다 RAG로 관련 정보만 선택적으로 주입하는 것이 안전하다.

## 2편 정리 — L2에서 L3로

컨텍스트 엔지니어링(L2)이 다루는 것:
- **RAG**: Naive → Advanced → Modular → Agentic 4단계 진화
- **메모리 시스템**: In-context / External / In-weights 세 계층 + 대화 히스토리 관리
- **컨텍스트 창 관리**: Lost in the Middle, Stuffing vs RAG 선택, Prompt Caching, 압축

L2로 해결하기 어려운 것:
- 구조화된 외부 시스템 호출 (API, 데이터베이스)
- 실시간 데이터 접근
- 코드 실행, 파일 조작 등 실제 액션

이 한계를 다루는 것이 L3, 도구 통합이다. 3편에서는 Function Calling, MCP, Structured Output을 다룬다.
