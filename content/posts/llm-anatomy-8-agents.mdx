지금까지 이 시리즈는 LLM이 **생각하는 방법**을 따라왔다. Transformer 아키텍처, 정렬, 추론, 긴 문맥, 오픈소스 생태계, 아키텍처 진화, 그리고 추론 심화까지.

8편은 전환점이다. 생각에서 **행동**으로.

LLM이 텍스트를 예측하는 것을 넘어 외부 도구를 호출하고, 웹을 탐색하고, 코드를 실행하고, 수십 스텝에 걸친 태스크를 수행하는 에이전트로 진화할 때 무슨 일이 일어나는가. 그 기술적 메커니즘을 들여다본다.

## ReAct — 생각과 행동을 교차하다

### 왜 필요한가

에이전트 연구 초기에는 두 접근이 공존했다. 하나는 Chain-of-Thought(CoT): 모델이 내부적으로 긴 추론 흔적을 만들어 복잡한 문제를 푸는 방식. 강력하지만 외부 세계와 단절돼 있다. 모델이 어떤 사실을 모르거나 틀려도 수정할 방법이 없다.

다른 하나는 Act-only: 명령을 받아 외부 도구를 실행하는 방식. 추론 없이 행동만 하니 오류가 발생해도 "왜 틀렸는지" 파악하지 못하고 같은 실수를 반복한다.

ReAct는 이 둘을 통합한다.

### 핵심 아이디어 (Yao et al., 2022)

**Thought → Action → Observation** 사이클을 교차 반복한다. 한 스텝에서 모델은 자연어로 추론하고(Thought), 외부 환경에 행동을 실행하며(Action), 결과를 관찰(Observation)한 뒤 다시 추론한다. 추론 흔적이 행동 계획 수정, 예외 처리, 자기 교정의 근거가 된다.

```
Thought: 파리의 인구가 필요하다.
Action: Search[Paris population]
Observation: 파리의 도시 인구는 약 210만 명이다.
Thought: 이제 질문에 답할 수 있다.
Action: Finish[210만 명]
```

<img src="/images/llm-anatomy/react-loop.svg" alt="ReAct 루프 다이어그램 — Thought, Action, Observation이 순환하며 외부 도구 호출과 내부 추론을 결합" />

핵심은 **추론 흔적이 단순 메모가 아니라는 점**이다. 도구 호출 전에 "왜 이 도구를 쓰는가"를 명시하고, 결과를 받은 후 "이 결과가 내 가설과 어떻게 맞는가"를 재평가한다. 오류 복구가 가능해지는 이유다.

### 결과

Yao et al.(2022)의 실험에서 ReAct는 텍스트 게임 환경(ALFWorld)에서 imitation/RL 방식으로 수천~수십만 예시를 훈련한 기존 방법 대비 **34% 절대 성공률 향상**을 달성했다.<Ref id={1} /> 웹 쇼핑 에이전트(WebShop)에서도 RL 훈련 기반 대비 **10% 절대 향상**이었다. 최적 성능은 ReAct + CoT 결합이었다. 내부 지식과 외부 정보를 모두 활용하기 때문이다.

ReAct는 오늘날 대부분의 에이전트 시스템에서 기초 패턴으로 쓰인다. 이름이 드러나지 않더라도, "생각하고 → 도구 쓰고 → 결과 보고 → 다시 생각하는" 루프는 어디서나 등장한다.

## Tool Use / Function Calling — 도구를 쓰는 법을 학습하다

### 왜 필요한가

LLM 자체는 계산기가 없다. 최신 뉴스를 모른다. 외부 데이터베이스를 조회하지 못한다. 도구 없이는 모델의 지식은 훈련 시점에서 동결돼 있고, 수치 계산 오류가 빈번하다.

외부 도구를 호출하는 능력은 이 한계를 구조적으로 돌파한다.

### Toolformer: 모델이 스스로 배우다 (Schick et al., 2023)

초기 접근 중 하나는 모델에게 도구 사용법을 자기지도(self-supervised) 방식으로 학습시키는 것이었다.<Ref id={2} /> Toolformer의 아이디어: 모델이 각 도구에 대한 소수의 시연(few demonstrations)만으로, API를 어디서 호출할지, 언제 호출할지, 어떤 인자를 넘길지를 스스로 결정하도록 훈련한다.

지원 도구는 계산기, Q&A 시스템, 검색엔진, 번역기, 달력이었다. 결과는 주목할 만했다. 도구를 활용한 훨씬 작은 모델이 도구 없는 훨씬 큰 모델과 경쟁했고, 언어 모델링 능력도 유지했다.

### 현대적 Function Calling: JSON Schema 기반 표준화

오늘날 OpenAI, Anthropic, Google이 채택한 방식은 JSON Schema 기반 표준화다. 구조는 네 요소로 구성된다.

1. **name** — 도구 고유 식별자
2. **description** — 언제/왜 사용하는지 의미론적 설명
3. **parameters** — JSON Schema로 정의된 입력 구조
4. **required** — 필수 필드 목록

동작 원리는 다음과 같다.

1. 모델이 description을 보고 도구 선택 결정
2. 모델이 parameters 스키마에 맞는 JSON 출력 생성
3. 호스트 애플리케이션이 실제 함수를 실행하고 결과를 모델 컨텍스트에 삽입
4. 모델이 결과를 바탕으로 응답 또는 추가 호출

한 가지 중요한 사실: **description 품질이 도구 선택 정확도의 핵심 변수다**. 스키마가 아무리 잘 정의돼 있어도 설명이 모호하면 모델이 올바른 도구를 선택하지 못한다. "이 도구는 어떤 상황에서, 왜 써야 하는가"를 설명하는 것이 파라미터 타입 정의만큼이나 중요하다.

## MCP — 도구 통합의 표준화

### M×N 문제

에이전트 생태계가 성숙하면서 새로운 엔지니어링 문제가 등장했다. M개의 LLM 애플리케이션과 N개의 도구/데이터소스가 있을 때, 각각을 개별 커스텀으로 연결하면 **M×N개의 통합**이 필요하다. 각 통합은 별도 코드, 인증, 데이터 형식, 에러 처리를 요구한다. 유지보수 비용이 조합적으로 폭발한다.

MCP(Model Context Protocol)는 이 문제를 **M+N**으로 줄이는 표준 프로토콜이다.

### 핵심 아이디어

Anthropic이 2024년 11월 오픈소스로 공개한 JSON-RPC 2.0 기반 LLM-도구 통신 표준.<Ref id={3} /> 세 계층으로 구성된다.

- **MCP Host**: 사용자와 상호작용하는 AI 애플리케이션 (Claude Desktop, IDE 등). LLM을 내부에 포함.
- **MCP Client**: Host 내부에 위치. LLM의 요청을 MCP 메시지로 변환하고 서버 응답을 번역.
- **MCP Server**: 외부 도구/데이터소스 앞단의 독립 프로세스. 세 기능을 제공한다.
  - **Resources**: 모델에 컨텍스트로 제공되는 구조화된 데이터
  - **Tools**: 모델이 실행할 수 있는 함수/액션
  - **Prompts**: 사전 정의된 템플릿/지시문

통신 방식은 두 가지다. 로컬은 stdio(표준 입출력, 빠른 동기 전송), 원격은 SSE(Server-Sent Events, 실시간 스트리밍).

<img src="/images/llm-anatomy/mcp-architecture.svg" alt="MCP 아키텍처 — Host, Client, 여러 MCP Server가 JSON-RPC 2.0으로 통신하는 M+N 통합 구조" />

중요한 기능 하나: **동적 능력 탐색(Dynamic Capability Discovery)**. 에이전트가 MCP 서버에 런타임에 "어떤 도구와 스키마가 사용 가능한가"를 물어볼 수 있다. 새 도구를 추가할 때 모델 재훈련이 필요 없다.

### 생태계 수렴

2024년 11월 Anthropic 공개 이후, OpenAI와 Google DeepMind가 채택했다. 2025년 12월에는 Linux Foundation 산하 Agentic AI Foundation(AAIF)에 기증됐다. Anthropic, Block, OpenAI가 공동 설립하고, Google, Microsoft, AWS, Cloudflare, Bloomberg이 지원하는 중립 거버넌스로 이동했다.

"어떻게 연결하는가"의 표준이 수렴하고 있다는 신호다.

## Computer Use — API 없는 세계를 탐색하다

### 왜 필요한가

Function Calling은 API가 있는 서비스에서만 작동한다. 그런데 세상의 대부분 소프트웨어는 API 없이 GUI로만 접근 가능하다. 스프레드시트, 레거시 기업 시스템, 데스크톱 앱, 심지어 최신 웹 서비스도 상당수가 API를 노출하지 않는다.

**인간처럼 화면을 보고 마우스/키보드로 조작하는 능력**이 범용 에이전트의 핵심 병목이었다.

### 공통 루프

구현 방식에 상관없이 Computer Use 에이전트의 기본 루프는 동일하다.

```
스크린샷 수신 → 비전으로 상태 파악 → 다음 행동 결정
→ 마우스/키보드 액션 실행 → 스크린샷 재수신 → 반복
```

상태를 텍스트 대신 픽셀로 인코딩하고, 행동을 API 호출 대신 GUI 이벤트로 출력한다는 것이 핵심 차이다.

### Anthropic의 접근 (2024년 10월 베타)

Anthropic의 공식 블로그("Developing a Computer Use Model")에 따르면,<Ref id={4} /> 모델은 화면의 특정 위치를 클릭하기 위해 수직·수평으로 이동해야 할 픽셀 수를 직접 계산한다. 해상도나 DPI 스케일링과 무관하게 작동하는 방식이다. 백엔드는 Xvfb(X Virtual Framebuffer) 등 가상 디스플레이 기술을 활용한다.

현재 한계도 공개돼 있다. 드래그, 줌 등 일부 복잡한 동작은 지원하지 않고, 연속 스트림이 아닌 "플립북" 방식으로 화면을 보기 때문에 짧은 알림 창처럼 빠르게 나타났다 사라지는 요소를 놓친다.

### OpenAI CUA (2025년 1월)

OpenAI의 Computer-Using Agent는 GPT-4o 비전과 RL 기반 추론을 결합했다. 클라우드 기반 가상 브라우저 환경에서 실행되며, 로그인이나 CAPTCHA 같은 민감한 액션에서는 사용자 확인을 요청한다.

OpenAI 공식 발표 기준 성능 수치다.<Ref id={5} />

| 벤치마크 | 성능 |
|---------|------|
| OSWorld (전체 컴퓨터 사용) | 38.1% |
| WebArena (웹 기반 태스크) | 58.1% |
| WebVoyager | 87.0% |

OSWorld에서 38.1%라는 수치는 절대적으로 높지 않다. 컴퓨터 사용이 얼마나 어려운 태스크인지를 방증하기도 한다. 동시에 불과 수년 전과 비교하면 이 영역에서 AI가 사실상 불가능했음을 감안하면 의미 있는 진전이다.

## 코딩 에이전트 — 실행 환경과 반복 루프

에이전트가 가장 빠르게 실용화된 영역 중 하나가 소프트웨어 개발이다. 코딩은 명확한 입출력(코드 실행 결과)과 자동화된 검증(테스트)이 있어 에이전트가 피드백 루프를 돌리기 좋은 환경이다.

### 공통 기술 아키텍처

제품마다 구현이 다르지만, 코딩 에이전트의 공통 패턴은 세 요소로 수렴한다.

**1. 샌드박스 환경**: Shell(명령 실행), 코드 편집기, 브라우저, 지속적 워크스페이스. 멀티 스텝 실행 중 상태(파일, 변수, 환경변수)를 유지한다.

**2. 계획-실행-검증 루프**: 요구사항 분석 → 코드 생성 → 실행/테스트 → 오류 분석 → 수정 반복. 수천 스텝의 긴 호라이즌 태스크를 처리한다.

**3. 컨텍스트 관리**: 코드베이스 전체를 탐색하고, 관련 파일을 식별하며, 레포지토리 구조를 파악하는 능력. 단순 코드 생성이 아니라 **기존 코드베이스를 이해하고 수정하는 것**이 핵심이다.

### SWE-bench: 코딩 에이전트의 척도

SWE-bench는 실제 GitHub 이슈를 해결하는 능력을 측정하는 벤치마크다. "unassisted" 모드에서 모델은 관련 파일 목록 없이 스스로 코드베이스를 탐색해야 한다.

진화 속도가 눈에 띈다.

| 시점 | 성능 |
|------|------|
| 2024년 3월 (Devin 발표) | 13.86% (이전 SOTA: 1.96%) |
| 2026년 3월 현재 | Claude Opus 4.5 80.9% (SWE-bench Verified 리더보드 기준) |

약 2년 사이 1.96% → 80.9%.<Ref id={6} /> SWE-bench Verified 리더보드는 지속적으로 갱신되므로, 이 수치는 집필 시점 기준이다.

단, 이 수치를 그대로 받아들이기 전에 한 가지 맥락이 중요하다. SWE-bench Pro(더 어려운 변형)에서는 상위 모델들도 23%대에 머문다. "실험실 벤치마크와 실제 코딩 태스크의 복잡도 차이"가 아직 크다는 신호다.

### 2025년의 패턴: Planner + Executor + Verifier

2025년 기준 상위 코딩 에이전트 시스템 대부분은 단일 에이전트가 아닌 **Planner + Executor + Verifier** 구조를 취한다. 계획을 세우는 에이전트, 코드를 작성하는 에이전트, 검증하는 에이전트가 분리된다. 단일 에이전트 대비 복잡한 장기 태스크에서 확장성이 좋다는 것이 실험적으로 확인되고 있다.

## 멀티 에이전트 시스템 — 협업의 아키텍처

### 왜 필요한가

단일 에이전트는 구조적 한계를 갖는다. 컨텍스트 창이 유한하다. 병렬 작업이 불가능하다. 단일 실패점이 전체 태스크를 중단시킨다. 복잡한 전문 지식이 필요한 영역에서 역할 분화가 없다.

복잡한 태스크에서 여러 전문화된 에이전트가 협업하는 **멀티 에이전트 아키텍처**가 부상한 이유다.

### 핵심 아키텍처 패턴

**1. 오케스트레이터-서브에이전트 (계층형)**

하나의 오케스트레이터가 태스크를 분해하고 서브에이전트에 할당한다. 결과를 취합·조정한다. 현재 가장 널리 쓰이는 패턴이다. 오케스트레이터가 전략을 세우고 서브에이전트가 실행하는 구조는 인간 조직의 계층적 협업과 유사하다.

**2. 대화 기반 협업 (AutoGen 스타일)**

Wu et al.(2023)의 AutoGen 프레임워크는 에이전트들이 다중 턴 대화로 문제를 해결하는 구조를 제안했다.<Ref id={7} /> 각 에이전트는 역할, 시스템 프롬프트, 도구 접근 권한을 가진다. 통일된 메시징 인터페이스(send/receive/generate_reply)로 정적/동적 대화 패턴 모두를 지원한다.

AutoGen v0.4(2024)에서는 액터 모델 기반 비동기/이벤트 구동 아키텍처로 전환했다. OpenTelemetry 기반 관찰성(observability)과 분산 설계가 추가됐다. 실제 프로덕션 환경을 의식한 변화다.

**3. 반사형 아키텍처 (Reflexion)**

에이전트가 자신의 출력을 다른 역할(비평가, 검증자 등)로 재검토한다. 셀프 리뷰가 품질을 높인다. 인간의 "초안 → 검토 → 수정" 과정을 에이전트 내부에 구현하는 방식이다.

**4. 전문화 + 라우팅**

입력 특성에 따라 적합한 전문 에이전트로 라우팅한다. 코딩, 검색, 수학, 법률 등 도메인별 전문화 에이전트가 협업한다.

### 설계 원칙

멀티 에이전트 시스템에서 반복적으로 등장하는 설계 고려사항들이다.

- **느슨한 결합(Loose Coupling)**: 에이전트들이 표준 메시지 인터페이스를 통해 통신하고, 내부 구현은 독립적으로 유지한다. 개별 에이전트를 교체하거나 업그레이드할 때 다른 에이전트에 영향이 없어야 한다.
- **컨텍스트 공유 전략**: 전체 히스토리를 공유할 것인가, 요약을 전달할 것인가, 구조화된 메모리를 쓸 것인가. 컨텍스트 창 비용과 정보 손실 간의 트레이드오프다.
- **인간 인더루프(Human-in-the-Loop)**: 특정 결정 지점에서 인간 승인을 요청하는 설계. 자율성과 안전성 간의 균형을 조율하는 메커니즘이다.

## 에이전트의 한계와 과제

에이전트 기술에 대한 서술이 능력 강조로만 끝나면 절반의 그림이다. 현재의 한계가 더 중요한 경우가 많다.

### 안전성은 아직 초기

Agent-SafetyBench(Liu et al., 2024, arXiv:2412.14470)<Ref id={8} />는 16개 주요 LLM 에이전트를 평가했다. 결과: **안전 점수 60%를 초과한 에이전트가 없었다**. 두 가지 근본 결함이 확인됐다.

첫째, 견고성 부족. 예상치 못한 입력이나 환경에서 에이전트의 행동이 일관적이지 않다. 둘째, 위험 인식 부족. 위험한 행동의 결과를 선제적으로 인식하지 못한다.

"제어 가능성 딜레마"라는 문제도 있다. 안전 제약이 태스크 성능을 저하시키는 "준수 비용(Cost of Compliance)"이 존재한다. 반대로 "준수의 환상(Illusion of Compliance)"도 관측된다. 에이전트가 규칙을 따르는 것처럼 보이지만 실제로는 규칙 준수가 태스크 무능력을 가리는 경우다.

### 장기 계획 문제

긴 호라이즌 태스크에서 각 스텝의 소오류가 복합적으로 누적된다. 수십~수백 스텝에 걸친 컨텍스트와 상태를 유지하는 것 자체가 어렵다. 쉬운 태스크는 잘 해결하지만 실제 복잡한 태스크에서는 성능이 급격히 저하된다 — SWE-bench Verified 80%+ vs SWE-bench Pro 23%대의 간극이 이를 보여준다.

### 현실적인 배포 원칙

현재 시점에서 신뢰할 수 있는 에이전트를 만들기 위한 실용적 원칙들이 수렴되고 있다.

1. **최소 권한**: 태스크에 필요한 최소한의 도구와 권한만 부여한다.
2. **인간 인더루프**: 파일 삭제, 이메일 발송처럼 돌이킬 수 없는 행동 전에 승인을 요청한다.
3. **샌드박스 격리**: 에이전트 실행 환경을 프로덕션과 격리한다.
4. **감사 가능성**: 모든 행동을 로그로 기록하고 추적할 수 있어야 한다.

에이전트가 강력할수록 이 원칙들의 중요성도 커진다. "에이전트가 무엇을 할 수 있는가"보다 "에이전트가 무엇을 하지 않도록 설계하는가"가 실제 배포에서 더 많은 공학적 노력을 요구한다.

## 정리: 행동하는 모델의 구조

LLM이 에이전트로 진화하는 과정에는 뚜렷한 기술적 계층이 있다.

ReAct 패턴이 **추론과 행동의 결합** 방법을 정립했다. Function Calling이 **외부 도구를 호출하는 표준 인터페이스**를 만들었다. MCP가 **도구 통합의 표준화**를 진행 중이다. Computer Use가 **API 없는 세계로의 접근**을 열었다. 멀티 에이전트 아키텍처가 **단일 에이전트의 한계를 협업으로 넘는 구조**를 제시한다.

각 계층은 독립적이지 않다. ReAct 없이는 도구 호출이 맹목적이 된다. MCP 없이는 도구가 늘어날수록 통합 비용이 폭발한다. 멀티 에이전트 없이는 복잡한 장기 태스크의 확장이 어렵다.

9편은 이 모든 흐름이 수렴하는 지점을 들여다본다. 멀티모달, 스케일링 법칙의 재검토, 오픈소스의 부상, 그리고 AGI 논의까지.
