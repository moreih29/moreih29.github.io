LLM을 "그냥 배포"하면 무슨 일이 벌어질까.

ResNet이나 BERT를 서빙할 때 썼던 방식 — 배치를 묶고, 모델을 올리고, 요청이 오면 forward pass — 을 GPT 계열 모델에 그대로 적용하면 GPU는 대부분의 시간을 놀린다. 처리량은 기대의 수십 분의 일에 그치고, 메모리는 요청이 조금만 길어져도 폭발한다.

이 글은 그 이유를 먼저 짚고, 그 문제를 해결하기 위해 등장한 핵심 기술들을 하나씩 해부한다.

---

## 이 글의 위치

"LLM 서빙" 시리즈는 3편으로 구성된다.

- **1편 (이 글)**: 핵심 기술 — LLM 추론의 본질적 어려움과 이를 극복하는 알고리즘들
- **2편**: 프레임워크 — vLLM, SGLang, TensorRT-LLM 등 실제 구현체들의 특성과 선택 기준
- **3편**: 생태계 — 프로덕션 배포 전략, 하드웨어 조합, 모니터링과 운영

기술을 먼저 이해해야 프레임워크 선택이 의미 있고, 프레임워크를 알아야 생태계가 읽힌다. 1편에서 시작하는 이유다.

## LLM 추론은 왜 다른가

### Autoregressive 생성의 본질

일반 분류 모델은 입력 → 한 번의 forward pass → 출력이다. 병렬화가 쉽고, 배치 크기를 늘리면 GPU 활용률이 올라간다.

LLM은 다르다. 텍스트를 **한 토큰씩, 순차적으로** 생성한다. 토큰 t를 생성하려면 이전 토큰 0~t-1이 모두 있어야 한다. 이 자기회귀(autoregressive) 특성 때문에 디코딩은 본질적으로 직렬이다.

### Prefill은 compute-bound, Decode는 memory-bound

LLM 추론은 두 단계로 나뉜다.

**Prefill**: 입력 프롬프트 전체를 처음 처리하는 단계. 수백~수천 토큰을 병렬로 처리하므로 행렬 곱 연산이 집약적이다. GPU의 Tensor Core를 잘 활용할 수 있어 compute-bound다. 이 단계가 끝나야 첫 토큰이 나온다(TTFT, Time to First Token).

**Decode**: 토큰을 한 개씩 생성하는 단계. 매 스텝마다 이전 토큰들의 Key/Value 행렬을 KV 캐시에서 읽어야 한다. 읽는 데이터양(KV 캐시 크기)에 비해 실제 연산량이 적어 memory-bound다. GPU는 빠른데 메모리 대역폭이 병목이다. 토큰 간 평균 생성 시간(TPOT, Time per Output Token)이 여기서 결정된다.

이 두 단계의 특성 차이가 서빙 설계를 복잡하게 만드는 근본 원인이다.

### KV 캐시의 메모리 폭발

재계산 없이 이전 토큰들의 Key/Value 행렬을 저장하는 KV 캐시는 컨텍스트 길이에 비례해 커진다. Llama 3 70B 모델을 16비트 정밀도로 서빙한다면, 시퀀스 길이 8192 기준 단일 요청의 KV 캐시만 수십 GB에 달할 수 있다. 요청이 동시에 수십 개씩 들어오면 GPU 메모리가 순식간에 고갈된다.

그리고 기존 시스템들은 이 메모리를 최대 길이 기준으로 미리 예약(pre-allocation)했기 때문에, 실제 사용량과 무관하게 60~80%가 낭비됐다.

## 핵심 기술 1: 메모리 최적화

### PagedAttention — OS 페이징을 KV 캐시에

2023년 vLLM 논문이 제안한 PagedAttention은 운영체제의 가상 메모리 페이징 개념을 KV 캐시 관리에 적용했다<Ref id={1} />. KV 캐시를 고정 크기 블록(기본 16 토큰)으로 분할하고, 논리적 블록 번호와 물리적 GPU 메모리 위치 사이의 매핑 테이블을 유지한다. 모델은 연속적 메모리를 보는 것처럼 동작하지만, 실제로는 GPU 메모리 곳곳에 분산 배치된다<Ref id={2} />.

결과: KV 캐시 메모리 낭비가 60~80%에서 4% 미만으로 감소했다. 동일 레이턴시에서 처리량 2~4배 향상이 보고됐다.

공유 시스템 프롬프트를 쓰는 요청들은 같은 물리 블록을 참조한다. 쓰기가 발생할 때만 블록을 복사하는 Copy-on-Write 방식이다.

![PagedAttention 메모리 레이아웃 — 요청 A와 B가 각각의 논리 블록을 가지지만 블록 테이블을 통해 흩어진 물리 블록에 매핑된다. Phys #3처럼 공통 접두어를 담은 물리 블록은 두 요청이 공유하여 메모리를 절약한다](/images/llm-serving/paged-attention.svg)

### Prefix Caching — 공통 앞부분을 재사용

시스템 프롬프트나 Few-shot 예시처럼 여러 요청에서 반복되는 토큰 시퀀스는 매번 prefill을 다시 할 필요가 없다. 이미 계산된 KV 블록을 재사용하면 된다.

구현 방식은 두 가지다.

**vLLM의 블록 해싱**: KV 블록 단위로 해시를 계산해 동일 블록을 캐시 풀에서 조회한다. 구현이 단순하지만 블록 경계에 맞지 않는 접두어는 재사용이 안 된다<Ref id={3} />.

**SGLang의 RadixAttention**: Radix Tree 자료구조를 유지하며 LRU 정책으로 KV 캐시를 자동으로 발견하고 재사용한다. 다양한 멀티턴 대화 패턴에서 vLLM 대비 약 10% 처리량 우위를 보인다<Ref id={4} />.

수천 토큰짜리 시스템 프롬프트를 공유하는 RAG 파이프라인이나 코드 에디터 에이전트에서 특히 효과적이다.

## 핵심 기술 2: 배치 전략

### Static → Dynamic → Continuous

배치 전략의 진화는 "GPU를 얼마나 놀리지 않는가"의 역사다.

**Static Batching**: 고정 크기 배치를 구성하고, 배치 내 모든 요청이 끝날 때까지 기다린다. 짧은 요청이 먼저 끝나도 긴 요청이 완료될 때까지 GPU는 부분적으로 유휴 상태다. 가변 길이 출력이 일상인 LLM에서는 극히 비효율적이다<Ref id={5} />.

**Dynamic Batching**: 시간 창(time window)을 설정해 그 시간 내 도착한 요청들을 묶는다. 배치 크기가 유연해졌지만, 여전히 배치 경계에서만 요청을 교체한다.

**Continuous Batching(= Iteration-Level Scheduling)**: 매 디코딩 스텝(iteration)마다 배치 구성을 재평가한다. 어떤 요청이 완료되면 즉시 대기열의 새 요청으로 자리를 채운다. ORCA 논문에서 "iteration-level scheduling"으로 처음 제안한 방식이다<Ref id={6} />.

Anyscale 실험에서는 동일 워크로드 대비 **처리량 23배** 향상이 보고됐다<Ref id={7} />. 현재 vLLM, SGLang, TensorRT-LLM, TGI 모두 Continuous Batching을 기본으로 채택한다.

## 핵심 기술 3: 양자화

양자화는 가중치를 낮은 비트 표현으로 압축해 메모리와 연산 비용을 줄인다. 방법마다 특성이 다르다.

### GPTQ

Post-Training Quantization(PTQ). 레이어별로 역헤시안 행렬을 이용한 2차 최적화로 4비트 양자화를 수행한다<Ref id={8} />. Calibration 데이터셋이 필요하고, 데이터셋 품질에 결과가 의존한다. FP16 대비 메모리 4배 절감이 가능하고, Marlin 커널 적용 시 오히려 FP16보다 빠른 712 tok/s를 달성한다.

### AWQ (Activation-aware Weight Quantization)

활성화 분포를 분석해 중요한 가중치(salient weights)를 보호하는 방식으로 4비트 양자화를 수행한다. 역전파가 불필요하고 캘리브레이션 데이터 요구량이 GPTQ보다 적다<Ref id={9} />. Marlin-AWQ 커널 기준 741 tok/s로 조사된 방법 중 가장 빠른 처리량을 보이며, FP16 품질의 약 92~95%를 유지한다.

### GGUF

llama.cpp 생태계의 파일 포맷. Q2~Q8까지 다양한 비트폭을 지원하고, CPU 추론 또는 CPU+GPU 혼합 실행에 최적화돼 있다. Q4\_K\_M 기준 퍼플렉서티가 AWQ보다 약간 우수한 경우도 있다<Ref id={10} />. GPU 서버가 아닌 로컬 환경이나 Apple Silicon에서는 사실상 유일한 실용적 선택이다.

### FP8

8비트 부동소수점 포맷. NVIDIA Hopper(H100) 이상에서 하드웨어 가속이 지원된다<Ref id={11} />. FP16 대비 메모리 2배 절감, 처리량 최대 1.6배 향상이 가능하다. H100 Transformer Engine을 활용하면 FP16 대비 최대 33% 빠른 결과가 보고됐다<Ref id={12} />. 차세대 Blackwell(B200)에서는 FP4까지 지원 범위가 확장됐다.

### 선택 기준 요약

| 방법 | 환경 | 특징 |
|------|------|------|
| AWQ | NVIDIA GPU 서버 | 최고 처리량, 캘리브레이션 최소 |
| GPTQ | NVIDIA GPU 서버 | 안정적, 캘리브레이션 데이터 필요 |
| FP8 | H100+ (Hopper) | 하드웨어 네이티브, 품질 손실 최소 |
| GGUF | CPU / 로컬 | CPU 실행 유일한 선택 |

## 핵심 기술 4: 병렬화

모델이 단일 GPU에 올라가지 않거나, 레이턴시를 더 낮춰야 할 때 병렬화가 필요하다. 전략마다 적합한 상황이 다르다.

### Tensor Parallelism (TP)

개별 레이어의 행렬을 여러 GPU에 열(column) 또는 행(row) 단위로 분할한다<Ref id={13} />. 각 GPU가 행렬의 일부를 보유하고 병렬 연산한 뒤, AllReduce 통신으로 결과를 합산한다. 단일 레이어가 여러 GPU에 걸쳐 실행되므로 레이턴시가 낮아진다. 단, GPU 간 통신 오버헤드가 있어 NVLink나 InfiniBand 같은 고속 인터커넥트가 중요하다.

### Pipeline Parallelism (PP)

모델 레이어 전체를 순차적 청크로 나눠 각 GPU에 할당한다. 4-way PP에서는 각 GPU가 레이어의 1/4을 담당한다. 각 장치가 이전 장치의 출력을 기다려야 하므로 "파이프라인 버블"이라는 유휴 시간이 발생한다. 마이크로배치(micro-batch) 기법으로 버블을 줄일 수 있지만 완전히 제거하기 어렵다.

### Sequence Parallelism (SP)

시퀀스 차원으로 분할해 긴 시퀀스 처리 시 TP와 함께 사용한다<Ref id={14} />. Meta에서는 이를 Context Parallelism(CP)이라 부른다. TP 영역과 SP 영역 사이 전환 시 AllGather/ReduceScatter 통신이 필요하다. 단일 GPU 메모리 한계를 넘는 긴 컨텍스트를 처리할 때 핵심 기법이다.

### Expert Parallelism (EP) — MoE 전용

Mixture-of-Experts 모델에서 각 expert를 서로 다른 GPU에 배치한다. 토큰은 라우터에 의해 해당 expert가 있는 GPU로 전송된다(All-to-All 통신)<Ref id={15} />. DeepSeek V3, Mixtral 같은 MoE 모델을 서빙할 때 필수다. 다만 All-to-All 통신이 end-to-end 레이턴시의 10~30%를 차지하고, expert 부하 불균형(hot/cold expert) 문제가 있다.

## 핵심 기술 5: 디코딩 최적화

### Speculative Decoding — 추측하고 검증

소형 draft 모델이 k개 토큰을 순차로 추측하고, 대형 target 모델이 단 한 번의 forward pass로 k+1개 토큰을 일괄 검증한다. 수락된 토큰은 채택하고, 거절된 위치부터 재생성한다<Ref id={16} />.

수학적으로 target 모델 단독 출력과 동일한 분포를 보장하므로 품질 손실이 없다. vLLM 통합 기준 최대 2.8배 처리량 향상이 보고됐다.

구현 방식도 다양하다. **Medusa**는 target 모델에 추가 "head"를 붙여 미래 토큰 K개를 동시 예측한다. **EAGLE** 시리즈(SafeAILab)는 더 발전된 방향을 추구했다. EAGLE-3(NeurIPS'25)은 저/중/고 레벨 시맨틱 feature를 융합해 draft 품질을 높여 2~6배 속도 향상을 달성했다<Ref id={16} />.

단, 고배치 환경에서는 draft 모델 실행 비용이 이득을 상쇄할 수 있다. 저지연/소배치 환경에서 효과가 가장 크다.

### Guided Decoding — 구조화된 출력 보장

각 디코딩 스텝에서 문법/스키마를 위반하는 토큰의 확률을 0으로 마스킹해 JSON, 정규식, Context-Free Grammar에 맞는 출력을 강제한다.

**XGrammar**(CMU/MLC)는 사전 계산된 토큰 마스크와 병렬 처리로 기존 대비 100배 가속을 달성했다. 2025년 vLLM의 기본 guided decoding 엔진이 됐다<Ref id={17} />. OpenAI의 Structured Outputs 기반 기술인 **llguidance**(Microsoft)는 2025년 vLLM v0.8.2에 통합됐다.

### Chunked Prefill — Prefill이 Decode를 방해하지 않도록

긴 프롬프트의 prefill을 작은 청크로 분할하여 decode 요청과 함께 배치한다<Ref id={18} />. 하나의 긴 prefill이 engine step을 독점해 다른 요청의 응답을 지연시키는 문제를 해결한다.

compute-bound(prefill)와 memory-bound(decode) 연산의 균형을 맞춰 대기 요청들의 TTFT 증가를 방지한다. SGLang에서는 Splitfuse라는 이름으로 구현돼 있다.

## 핵심 기술 6: 2025~2026 최신 기법

### FlashAttention-3 / 4 — Attention 커널의 진화

FlashAttention은 Attention 연산을 GPU SRAM에 맞게 타일링하여 HBM(고대역폭 메모리) 접근을 최소화하는 방식이다. FlashAttention-3(2024)은 NVIDIA Hopper(H100)의 비동기 처리 능력을 활용해 Tensor Core 연산과 메모리 이동을 오버랩한다<Ref id={19} />. BF16에서 840 TFLOPs/s(H100 활용률 75%), FP8에서 1.3 PFLOPs/s를 달성했다. FA-2 대비 1.5~2.0배 속도 향상이다.

FlashAttention-4(2025)는 Hopper와 Blackwell(B200) 모두를 대상으로 CuTeDSL로 재작성됐다<Ref id={20} />. softmax 출력 재스케일링의 90%를 생략하는 새 온라인 softmax 알고리즘을 도입했고, Blackwell에서 NVIDIA cuDNN attention 커널보다 최대 22% 빠른 결과가 보고됐다.

### Disaggregated Inference — Prefill과 Decode를 분리

Prefill(compute-bound)과 Decode(memory-bound)를 서로 다른 물리 장치 집합으로 분리하여 각각 독립적으로 스케일링하는 아키텍처다. DistServe(OSDI'24)가 이론적 프레임워크를 제시했고<Ref id={21} />, 2025년에는 NVIDIA Dynamo, llm-d, SGLang, vLLM, Mooncake 등이 이를 기본 아키텍처로 채택하며 주류화됐다<Ref id={22} />.

혼합 배치에서 긴 prefill이 decode 레이턴시(TTFT)를 악화시키는 문제를 구조적으로 분리해 해결한다. 단, prefill/decode 워커 비율을 잘못 설정하면 goodput이 급락할 수 있어 운영 복잡도가 높다.

### CUDA Graph + torch.compile — CPU 오버헤드 제거

**CUDA Graph**는 GPU 커널 실행 그래프를 한 번 캡처하고, 이후 재실행 시 CPU→GPU CUDA API 호출 오버헤드를 제거한다<Ref id={23} />. vLLM V1은 Attention 연산처럼 CUDA Graph와 호환되지 않는 부분을 제외한 구간만 선택적으로 캡처하는 Piecewise CUDA Graphs를 도입했다.

**torch.compile**은 PyTorch JIT 컴파일러가 코드를 분석하여 최적화된 CUDA 커널을 자동 생성한다<Ref id={24} />. vLLM V1(2025)에서 기본으로 통합됐다. 하드웨어별 커널을 수동으로 최적화하지 않아도 배치 크기에 특화된 컴파일로 추가 성능을 얻는다.

## 정리 — 문제와 해결의 지도

LLM 서빙의 어려움은 본질적으로 두 가지 병목에서 온다. Prefill은 compute-bound이고, Decode는 memory-bound다. 이 비대칭이 모든 핵심 기술의 출발점이다.

| 문제 | 핵심 기술 | 효과 |
|------|-----------|------|
| KV 캐시 메모리 낭비 | PagedAttention | 낭비 60~80% → 4% 미만 |
| 반복 prefill 비용 | Prefix Caching | 공통 접두어 재사용 |
| GPU 유휴 시간 | Continuous Batching | 처리량 최대 23배 향상 |
| 모델 크기 | 양자화(AWQ/GPTQ/FP8) | 메모리 2~4배 절감 |
| 단일 GPU 한계 | TP/PP/SP/EP | 멀티 GPU 스케일링 |
| 직렬 decode 속도 | Speculative Decoding | 최대 2~6배 가속 |
| Prefill/Decode 간섭 | Disaggregated Inference | 독립적 스케일링 |
| Attention 커널 효율 | FlashAttention-3/4 | H100 활용률 35%→75% |

---

2편에서는 이 기술들이 실제 프레임워크에서 어떻게 구현됐는지를 다룬다. vLLM, SGLang, TensorRT-LLM, TGI — 각각 어떤 트레이드오프를 선택했고, 어떤 워크로드에 적합한지를 해부한다.
