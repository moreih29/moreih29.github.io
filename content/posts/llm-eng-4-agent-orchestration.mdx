"에이전트 하나에 모든 걸 시키면 되는 거 아닌가?"

처음에는 누구나 그렇게 생각한다. 그리고 대부분은 비슷한 순서로 벽에 부딪힌다. 컨텍스트가 넘쳐 잘리거나, 태스크가 너무 길어 집중력을 잃거나, 한 단계 실패가 전체를 망가뜨리거나.

에이전트 오케스트레이션(Agent Orchestration)은 그 문제에 대한 답이다. 단일 에이전트에 모든 책임을 주는 대신, 역할을 나누고 협력하게 만드는 것. 더 정확히는 — 오케스트라처럼, 각자 잘 하는 것을 하되 전체가 하나처럼 움직이게 하는 것.

이 글은 그 구조를 실무 관점에서 다룬다. 무엇이 있고, 언제 쓰고, 어디서 실패하는지.

---

## 이 글이 다루는 것

이 글은 L4, 에이전트 오케스트레이션이다. 1편(프롬프트 엔지니어링), 2편(컨텍스트 엔지니어링), 3편(도구 통합)을 거쳐 도착하는 지점이다.

L4가 다루는 질문은 이것이다. "여러 에이전트가 어떻게 협력해야 하는가?"

내부 메커니즘 — ReAct 패턴, Function Calling, Computer Use, 오케스트레이터-서브에이전트 기초, 코딩 에이전트 — 은 "LLM 해부학 8편"에서 이미 다뤘다. 이 글은 그 위에서 **어떤 프레임워크를 언제 쓰는가**, **어떤 패턴으로 협력하는가**, **어디서 실패하고 어떻게 대비하는가**를 중점으로 다룬다.

---

## 기원: 지능은 원래 분산되어 있다

에이전트 오케스트레이션 아이디어의 뿌리는 LLM 이전으로 거슬러 올라간다.

Marvin Minsky는 1986년 "Society of Mind"를 출판했다. 핵심 아이디어는 지능이 단일 중앙 시스템에서 나오는 것이 아니라, 각자 단순한 역할을 수행하는 수많은 "에이전트"들의 상호작용에서 창발한다는 것이었다. "에이전트"라는 용어가 오늘날과 유사한 의미로 사용된 초기 원천 중 하나다 [T3: Wikipedia].

같은 해, Rodney Brooks는 로보틱스에서 Subsumption Architecture를 발표했다. 단일 중앙 계획자 없이, 계층적으로 조직된 하위 행동들이 상향식으로 조합되어 지능적 행동을 만드는 구조였다 [T3: Wikipedia]. 중앙 집중식 오케스트레이션 대 분산 에이전트 — 이 논쟁의 역사적 시작점이다.

1990년대에는 JADE, BDI(Belief-Desire-Intention) 에이전트 아키텍처, FIPA 표준화 시도 등 멀티 에이전트 시스템 연구가 학술 영역에서 활발했다. 그러나 규칙 기반 로직의 한계로 현실적 문제 해결에는 도달하지 못했다. 학문의 영역에 머물렀다.

전환점은 LLM이었다.

## 전환점: LLM이 에이전트를 실용화하다

### LangChain의 등장 (2022년 말)

2022년 말 등장한 LangChain은 최초의 범용 LLM 에이전트 프레임워크였다. Tool 추상화, Memory 관리, Agent 루프를 표준화했다. 단순한 API 호출을 넘어, LLM이 도구를 쓰고 반복하는 루프를 처음으로 쉽게 구성할 수 있게 됐다.

이후 47M+ PyPI 다운로드 규모의 생태계로 성장한 것이 이 단계의 수요를 보여준다 [T3: Langfuse, 2025].

### AutoGen (2023)

Microsoft Research의 Wu et al.이 발표한 AutoGen [T1: arXiv:2308.08155]은 멀티 에이전트 대화(multi-agent conversation) 패러다임을 확립했다. 에이전트들이 다중 턴 대화로 문제를 해결한다. 통일된 메시징 인터페이스(send/receive/generate_reply)로 정적/동적 대화 패턴을 모두 지원하는 것이 핵심 설계였다.

### CrewAI (2023)

역할 기반(role-based) 에이전트 협업 프레임워크. 각 에이전트에 역할(role), 목표(goal), 배경(backstory)을 부여한다. LangChain 의존성 없이 독립 운영 가능하다는 것이 초기 채택을 이끌었다 [T2: CrewAI 공식 문서].

### Anthropic의 패턴 정리 (2024년 12월)

Anthropic이 "Building Effective Agents"를 공식 문서화했다 [T2: Anthropic, 2024]. 5가지 워크플로우 패턴을 정의하며 핵심 철학을 제시했다. "복잡한 프레임워크보다 단순한 조합 가능한 패턴." 이 문서는 이후 에이전트 설계 실무의 기준점이 됐다.

## 현재: 프레임워크 비교

2025~2026년 현재 주요 프레임워크들이 자리를 잡았다. 각각이 다른 강점을 가진다.

### LangGraph

LangChain 팀이 만든 그래프 기반 상태 머신이다. 노드(Node)와 엣지(Edge)로 워크플로우를 정의한다. 순환(cycle), 조건 분기, 병렬 실행, 영구 상태 관리를 지원한다.

StatefulGraph는 에이전트 실행 전반에 걸쳐 컨텍스트를 누적한다. 복잡한 분기 로직, 명시적 상태 관리가 필요한 장기 실행 워크플로우에 적합하다. 47M+ PyPI 다운로드 기준 가장 큰 커뮤니티를 가진다 [T2: LangChain, langchain.com/langgraph].

언제 쓸 것인가: 워크플로우에 분기와 루프가 복잡하게 얽혀 있고, 중간 상태를 명시적으로 추적해야 할 때. 코드로 그래프를 정의하는 방식이라 진입 비용이 있다.

### CrewAI (2025 현황)

Crews(협업 에이전트 팀)와 Flows(이벤트 구동 워크플로우)의 이원 구조다. 연구-초안 출판, 시장 분석, 컴플라이언스/QA에서 프로덕션 사례가 보고된다. 한 조직이 CrewAI로 특정 프로세스 단계에서 개발 시간 90% 단축을 보고했다 [T3: Latenode, 2025].

주목할 한계: 6~12개월 사용 후 커스텀 오케스트레이션 패턴이 필요해지면 LangGraph로 전환하는 케이스가 다수 있다 [T3: Latenode, 2025]. 빠른 시작에 강하지만, 복잡도가 올라가면 추상화가 제약이 될 수 있다.

언제 쓸 것인가: 역할이 명확한 팀 구조로 협업이 필요하고, 빠르게 프로토타입을 만들어야 할 때.

### AutoGen v0.4 (Microsoft, 2025년 1월)

완전한 재설계다. 액터 모델(Actor Model) 기반 비동기/이벤트 구동 아키텍처로 전환했다. 3계층 구조를 가진다.

- Core: 이벤트 구동 에이전트 시스템의 기반
- AgentChat: 그룹 채팅, 코드 실행, 사전 빌드 에이전트 등 고수준 API
- Extensions: Azure 코드 실행기, OpenAI 클라이언트 등 서드파티 통합

OpenTelemetry 기반 관찰성(observability)이 내장되어 있다. Magentic-One은 AutoGen v0.4 위에 구축된 제너럴리스트 멀티 에이전트 시스템으로, 오케스트레이터와 전문화된 에이전트 팀(FileSurfer, WebSurfer, Coder, ComputerTerminal)이 오픈 웹 및 파일 기반 태스크를 해결한다 [T2: Microsoft Research, 2025; T2: Microsoft AutoGen Blog, 2025].

언제 쓸 것인가: 분산 환경, 대규모 비동기 처리, Azure 인프라와 통합이 필요할 때.

### OpenAI Agents SDK (2025년 3월)

실험적 프레임워크였던 Swarm에서 프로덕션급으로 발전했다. 세 가지 핵심 기능이 있다.

**Handoffs**: 에이전트 간 작업 위임. 트리아지 에이전트가 입력을 분류해 전문 에이전트로 넘기는 패턴을 표준화한다.

**Guardrails**: 입출력 유효성 검사를 에이전트 실행과 병렬로 실행한다. 실행 흐름을 막지 않으면서 안전 검사를 한다.

**Tracing**: 기본 활성화. LLM 생성, 도구 호출, 핸드오프, 가드레일 이벤트 전체를 기록한다 [T2: OpenAI Agents SDK 공식 문서].

언제 쓸 것인가: OpenAI 모델 스택 위에서 빠르게 에이전트를 구성해야 할 때. 트리아지 패턴이 잘 들어맞는 경우.

### Claude Agent SDK (Anthropic, 2025년 9월)

원래 Claude Code SDK였으나 범용 에이전트 런타임으로 발전해 이름이 바뀌었다. 핵심 특징은 MCP 퍼스트(MCP-first) 설계다. MCP 서버와의 런타임 도구 탐색을 기본으로 지원한다.

Subagents는 병렬 실행되며 격리된 컨텍스트 창을 가지고, 결과만 오케스트레이터에 반환한다. Extended Thinking으로 추론 과정을 시각화해 디버깅과 신뢰 구축에 쓸 수 있다. Agent Skills는 전문 태스크용 동적 로드 가능 지식/스크립트 패키지다 [T2: Anthropic, "Building agents with the Claude Agent SDK"; T2: Claude Agent SDK 공식 문서].

5계층 아키텍처: MCP → Skills → Agent → Subagents → Agent Teams. Python/TypeScript 모두 지원한다.

언제 쓸 것인가: MCP 생태계를 활용하거나, Claude 모델 기반으로 서브에이전트 병렬 실행이 필요할 때.

## 워크플로우 패턴 — When/How to Use

프레임워크보다 더 근본적인 것이 패턴이다. 어떤 프레임워크를 써도 결국 아래 패턴들 중 하나이거나 그 조합이다.

<figure>
  <img src="/images/llm-eng/workflow-patterns.svg" alt="워크플로우 패턴 5종 비교 도식" />
  <figcaption>Sequential · Parallel · Router · Orchestrator-Workers · Evaluator-Optimizer — 5가지 핵심 패턴 구조</figcaption>
</figure>

Anthropic [T2: Building Effective Agents], Azure Architecture Center [T2], OpenAI Agents SDK [T2]의 자료를 교차 검증한 수렴 패턴이다.

### Sequential — Prompt Chaining

이전 단계 출력이 다음 단계 입력이 된다. 각 단계에서 프로그래매틱 게이트(gate) — 출력 검증 후 다음 단계 진행 — 를 두면 오류가 전파되지 않는다.

언제: 문서 추출 → 분류 → 요약처럼 단계가 명확하고 순서가 있는 태스크. 가장 단순하고 가장 이해하기 쉬운 패턴이다. 복잡한 패턴이 필요하다는 확신이 없다면 여기서 시작하라.

### Parallel — Sectioning & Voting

두 가지 변형이 있다.

Sectioning: 태스크를 독립 서브태스크로 분해해 병렬 실행, 결과를 집계한다. LangGraph에서는 Scatter-gather 패턴으로 구현한다.

Voting: 동일 태스크를 여러 에이전트가 독립적으로 실행하고 다수결로 최종 답을 선택한다. 단순히 여러 번 실행하는 것보다 신뢰도를 높이는 데 효과적이다.

언제: 독립적으로 처리 가능한 서브태스크가 있을 때(Sectioning), 또는 답의 신뢰도가 중요할 때(Voting).

### Router

입력 특성을 먼저 분류하고, 적합한 전문 에이전트나 프롬프트로 라우팅한다. 분해(decompose) → 라우팅(route) → 합성(synthesize)의 3단계다.

OpenAI Agents SDK의 Handoffs가 이 패턴을 표준화한 구현이다.

언제: 도메인별 처리 방식이 다른 다양한 입력을 처리할 때. 고객 서비스, 다언어 지원, 도메인별 전문 에이전트가 필요한 경우.

### Orchestrator-Workers

오케스트레이터가 태스크를 동적으로 분해하고, 워커를 선택해 위임하고, 결과를 취합한다. 워커들은 서로 직접 통신하지 않고 오케스트레이터에만 보고한다.

수렴된 베스트 프랙티스: 오케스트레이터 1개 + 전문화된 워커 2~4개. 이 비율이 조율 비용과 성능 사이의 균형점으로 실무에서 수렴됐다.

언제: 코딩(여러 파일 동시 수정), 연구(여러 소스 병렬 탐색), 복잡한 분석처럼 서브태스크가 사전에 정해지지 않고 동적으로 결정되는 경우.

### Evaluator-Optimizer

Generate → Evaluate → Improve 루프다. 평가자가 요구사항 대비 출력을 검토하고, 최적화자가 구체적 개선을 수행한다. 이것을 반복한다.

특정 성공 기준이 있지만 한 번에 충족되지 않을 때 유효하다.

언제: 문학 번역, 코드 리뷰 후 수정, 고품질 문서 생성. 단순 태스크에는 과잉이다. 루프 횟수에 상한을 반드시 두어야 한다.

## 멀티 에이전트 협업 — 현실적 평가

### 토론 기반 협업(MAD)의 솔직한 성과

Du et al.(2023) [T1: arXiv:2305.14325]이 제안한 Multi-Agent Debate(MAD)는 에이전트들이 독립적으로 초기 답변을 생성하고, 여러 라운드에 걸쳐 서로의 답변을 검토해 정제하는 방식이다. 이론적으로 매력적이다.

그런데 2025년 연구들의 평가는 냉정하다 [T2: ICLR Blogposts 2025]. MAD가 단순 단일 에이전트 전략(self-consistency, ensembling)을 일관적으로 능가하지 못한다. 성능 향상의 대부분이 Majority Voting으로 설명된다. 단, 이기종 모델 조합(heterogeneous agents) 시에는 더 뚜렷한 개선이 있다 — GSM-8K에서 91% vs 동일 모델 82%.

실무적 함의: 에이전트 토론을 구성하기 전에 단순한 Voting부터 시도하라. 복잡도 대비 성능 향상이 예상보다 낮을 수 있다. 이기종 모델 조합이라면 가능성이 높아진다.

### 계층형 오케스트레이션이 여전히 표준

Microsoft AutoGen의 Magentic-One, CrewAI의 Hierarchical Process가 공통적으로 계층형 오케스트레이션 구조를 취한다. 오케스트레이터가 전략, 워커가 실행. 이 분리가 실무에서 가장 안정적으로 작동하는 구조다.

## 실패 처리 — 어디서 깨지는가

멀티 에이전트 시스템이 실패하는 지점을 알아야 제대로 설계할 수 있다.

### 조정 실패가 가장 크다

Cemri et al.(2025)의 MAST 연구는 7개 오픈소스 프레임워크에서 1,642개 실행 추적을 분석했다 [T1: arXiv:2503.13657]. 결과가 명확하다. 멀티 에이전트 시스템의 최대 실패 범주가 조정 실패(Coordination Failure)로 전체의 36.94%를 차지한다. 세부 유형은 잘못된 가정으로 진행(6.80%), 태스크 이탈(7.40%), 추론-행동 불일치(13.2%) 등이다.

더 중요한 수치: 79%의 문제가 기술적 구현이 아닌 명세(specification)와 조정(coordination) 이슈에서 발생한다.

방지책: 모든 에이전트 쌍 사이에 명시적 입출력 계약(데이터 스키마)을 정의하라. 에이전트들이 무엇을 받고 무엇을 내보내는지 계약이 명확해야 한다.

### 재시도 전략

무한 재시도는 문제를 악화시킨다. 수렴된 실무 원칙들 [T3: fast.io; T3: sparkco.ai]:

- 지수적 백오프(Exponential Backoff) + 지터(Jitter): 여러 에이전트가 동시에 재시도해 시스템을 압도하는 Thundering Herd 문제를 방지한다.
- 최대 3회 재시도를 권장한다.
- 데드레터 큐(Dead-letter Queue): 재시도 한계를 초과한 태스크를 처리하는 별도 경로.

### Circuit Breaker 패턴

일관적으로 실패하는 도구나 API를 일시적으로 비활성화한다. 에이전트가 폴백 도구로 전환하거나 실행을 일시 중단한다. 한 컴포넌트의 실패가 전체 시스템으로 전파되는 연쇄 실패(Cascading Failure)를 방지하는 핵심 패턴이다.

### Human-in-the-Loop (HITL)

3계층 접근이 실무에서 수렴됐다 [T3: anyreach.ai]:

1. 저위험 불확실성: AI가 정제된 프롬프트로 재시도
2. 중위험: 티어-1 에이전트(인간 지원)로 즉시 이관
3. 고위험/컴플라이언스: 전문가 에스컬레이션

돌이킬 수 없는 행동 — 파일 삭제, 이메일 발송, 결제 — 직전에는 반드시 승인을 요청하라. 이것은 아키텍처 수준에서 설계해야 하는 것이지, 나중에 추가하는 것이 아니다.

### 사이클 방지

한 에이전트가 다른 에이전트를 트리거할 때 반드시 사이클 체크가 필요하다. 작게 보이는 루프가 무한 실행으로 이어질 수 있다.

## 평가 — 에이전트를 어떻게 측정하는가

에이전트 시스템을 구축했다면 측정할 수 있어야 한다. 주요 벤치마크들이다.

### GAIA

Mialon et al.(2023) [T1: arXiv:2311.12983]. 450개 질문으로 툴 사용, 멀티모달, 웹 탐색, 추론 등 종합 능력을 평가한다. 세 가지 난이도 레벨로 구성된다. HAL Leaderboard (hal.cs.princeton.edu/gaia)에서 지속 갱신된다.

### GAIA 2 (2025)

1,120개 시나리오. 동적/비동기 환경에서의 에이전트 평가. 스마트폰 환경(이메일, 메시징, 캘린더 등) 시뮬레이션을 포함하며, 에이전트 행동과 독립적으로 환경이 변화하는 시나리오도 있다 [T1: OpenReview, 2025].

### AgentBench

Liu et al.(2023) [T1: arXiv:2308.03688, ICLR 2024]. 멀티턴 오픈 엔드 설정에서 LLM을 에이전트로 평가한다. 추론 품질, 정확도, 멀티턴 일관성을 모두 측정한다.

### CUB (Computer Use Benchmark, 2025년 중반)

Theta AI 연구팀이 개발한 컴퓨터 사용 특화 벤치마크. 7개 산업 106개 엔드투엔드 워크플로우를 포함한다 [T2: o-mega.ai]. 비즈니스 운영, 금융, 이커머스 등 실제 업무 환경에 가까운 평가다.

코딩 에이전트 특화 SWE-bench는 "LLM 해부학 8편"에서 다룬다.

## 최신 동향 (2025~2026)

### 마이크로서비스 전환과 유사한 패턴

단일 범용 에이전트에서 오케스트레이션된 전문화 에이전트 팀으로의 전환이 진행 중이다. 2010년대 모놀리스 → 마이크로서비스 전환과 구조적으로 유사하다 [T2: machinelearningmastery.com]. 그때와 같은 교훈도 적용된다 — 마이크로서비스가 항상 모놀리스보다 좋은 것이 아닌 것처럼, 멀티 에이전트가 항상 단일 에이전트보다 좋지 않다.

### Agent-as-a-Service

에이전트를 서비스로 제공하는 플랫폼이 급증했다. Azure AI Foundry, AWS Bedrock AgentCore, Google Vertex AI 같은 기업 에이전트 플랫폼이 자리를 잡았다 [T2: Deloitte; T2: Google Cloud Blog]. 저코드/노코드 에이전트 빌더로 에이전트 구축에 15~60분이면 가능해졌다.

### 시장 수치와 현실

AI 에이전트 시장 CAGR 46.3%, 2025년 $7.84B → 2030년 $52.62B으로 여러 리포트가 유사한 수치를 보고한다 [T3: Landbase]. 2025년 기준 79% 조직이 에이전트형 AI를 일부 채택했다고 한다 [T3: Landbase]. Gartner에 따르면 Q1 2024 → Q2 2025 사이 멀티 에이전트 관련 기업 문의가 1,445% 급증했다 [T2: Gartner, 2025].

그러나 같은 Gartner가 예측한 것도 있다: 2027년까지 에이전트형 AI 프로젝트 40% 이상이 레거시 시스템 호환성 문제로 실패할 것이다 [T2: machinelearningmastery.com]. 채택이 빠를수록 현실적 한계와의 충돌도 빠르다.

### 에이전트 도입의 ROI

에이전트 도입 기업의 평균 ROI 171%, 미국 기업 192%로 전통 자동화의 3배라는 수치가 있다 [T2: Deloitte]. 숫자가 인상적이지만, 어떤 프로세스에서 어떻게 측정했는지가 중요하다. 숫자보다 자신의 도메인에서의 파일럿이 더 신뢰할 수 있는 데이터다.

## 4편 정리 — L4에서 L5로

에이전트 오케스트레이션(L4)이 다루는 것:

- 프레임워크 선택: LangGraph(복잡한 상태), CrewAI(역할 기반), AutoGen(비동기 대규모), OpenAI SDK(핸드오프), Claude Agent SDK(MCP 퍼스트)
- 워크플로우 패턴: Sequential, Parallel, Router, Orchestrator-Workers, Evaluator-Optimizer
- 실패 처리: 조정 실패 방지, 재시도 전략, Circuit Breaker, HITL
- 평가: GAIA, AgentBench, CUB

L4로 해결하기 어려운 것:

- 에이전트 시스템 전체가 의도대로 작동하는지 지속적으로 확인하는 것
- 런타임 환경 — 가드레일, 평가, 모니터링, 피드백 루프
- 실제 프로덕션에서 에이전트를 안전하게 운영하는 것

이것을 다루는 것이 L5, 하네스 엔지니어링이다. 5편에서는 Guardrails, 에이전트 평가, 모니터링, 피드백 루프를 다룬다.
