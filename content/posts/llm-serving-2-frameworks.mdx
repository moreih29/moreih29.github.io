1편에서 LLM 서빙의 본질적 어려움 — KV 캐시 메모리 폭발, Prefill/Decode 비대칭, 자기회귀 생성의 직렬성 — 과 이를 극복하는 핵심 기술들을 살펴봤다. 2편은 그 다음 질문으로 넘어간다: 이 기술들을 실제로 구현한 프레임워크들은 어떻게 다른가, 그리고 어떤 상황에서 무엇을 골라야 하는가.

---

## 이 글의 위치

- **1편 (이전 글)**: 핵심 기술 — PagedAttention, Continuous Batching, 양자화, Speculative Decoding, Disaggregated Inference 등
- **2편 (이 글)**: 프레임워크 — 11종 구현체의 특성, 벤치마크, 선택 기준
- **3편 (다음 글)**: 생태계 — 클라우드 플랫폼, 오케스트레이션, 배포 전략, 모니터링

기술 개념을 알아야 프레임워크 선택이 의미 있다. 이미 1편을 읽었다면 이 글의 내용이 훨씬 선명하게 읽힐 것이다.

## 프레임워크 지형도

2026년 현재 LLM 서빙 프레임워크는 크게 세 계층으로 나뉜다.

**1군 (프로덕션 고처리량)**: vLLM, SGLang, TensorRT-LLM. 수만 개 이상의 동시 요청을 처리하는 클라우드 서빙에 적합하다.

**2군 (로컬/엣지)**: llama.cpp, Ollama, LiteLLM. GPU 서버 없이도 LLM을 실행하거나, 여러 LLM 프로바이더를 하나의 인터페이스로 묶는 데 쓰인다.

**3군 (특수 목적)**: LMDeploy, DeepSpeed-MII, MLC LLM, Modular MAX. 특정 하드웨어, 특정 연구 생태계, 또는 모바일/브라우저 같은 특수 환경을 타깃으로 한다.

![LLM 서빙 프레임워크 포지셔닝 맵 — x축 설정 복잡도, y축 프로덕션 처리량, 버블 크기는 GitHub Stars. vLLM과 SGLang이 좌상단 고성능·저복잡도 영역에, TensorRT-LLM이 우상단 고성능·고복잡도 영역에 위치. llama.cpp와 Ollama는 하단 로컬/엣지 영역에, TGI는 점선 버블로 유지보수 모드를 표시](/images/llm-serving/framework-positioning-map.svg)

## 1군: 프로덕션 3강

### vLLM — 사실상의 업계 표준

PagedAttention의 원저자 팀이 만든 프레임워크다<Ref id={1} />. 2023년 SOSP에서 논문과 함께 공개된 이후 빠르게 업계 표준이 됐다.

2026년 4월 기준 GitHub Stars 75,400+<Ref id={2} />. 2025년에는 PyTorch 에코시스템에 공식 편입됐다<Ref id={3} />. 단일 리포지토리가 NVIDIA, AMD, Google, Meta, Intel 등 주요 AI 기업들의 기여를 모두 흡수하는 구조가 됐다.

**V1 아키텍처 (2025.01)**: 기존 대비 1.7배 속도 향상, 제로 오버헤드 프리픽스 캐싱, 강화된 멀티모달 지원을 담은 대규모 아키텍처 재설계다<Ref id={4} />. Piecewise CUDA Graphs로 CUDA Graph 호환성 문제를 해결했고, torch.compile을 기본 통합했다.

**vLLM Router (2025.12)**: Rust로 작성된 프로덕션급 로드 밸런서<Ref id={5} />. Prefill/Decode 분리(PD Disaggregation) 아키텍처를 지원하며, KV 캐시 상태를 인식한 지능형 라우팅이 가능하다.

지원 하드웨어가 가장 넓다. NVIDIA, AMD, Intel GPU, ARM CPU, PowerPC, TPU, Huawei Ascend까지 단일 코드베이스로 커버한다.

**언제 선택하는가**: 특정 하드웨어에 묶이고 싶지 않거나, 커뮤니티 지원이 중요하거나, 빠르게 프로덕션에 투입해야 할 때.

---

### SGLang — 구조화 생성과 캐시 효율의 강자

LMSYS Org(UC Berkeley + 산업계 협력)가 만든 프레임워크다. 핵심 기술은 **RadixAttention** — Radix Tree 자료구조로 KV 캐시를 관리해 멀티턴 대화, 시스템 프롬프트 공유, RAG 파이프라인처럼 접두어(prefix)가 반복되는 워크로드에서 자동으로 캐시를 재사용한다<Ref id={6} />.

2025년 말 기준 GitHub Stars 25,000+<Ref id={7} />. 전 세계 400,000+ GPU에서 운영 중이며, xAI, NVIDIA, Google Cloud, Microsoft Azure, AWS가 채택했다.

2025년 초 DeepSeek R1/V3 Day-1 지원으로 주목도가 급상승했다. NVIDIA GB200 NVL72에서 DeepSeek R1 기준 GPU당 26,000 입력 토큰/초를 달성했다<Ref id={8} />.

v0.4(2024.12)에서 추가된 Cache-Aware Load Balancer는 KV 캐시 재사용률을 고려해 요청을 라우팅한다. 구조화 출력(JSON 스키마, 정규식)도 기본 지원이다.

H100에서 ShareGPT 1,000 프롬프트 기준으로 vLLM 대비 처리량이 29% 우위를 보인다<Ref id={9} />. 멀티턴 워크로드에서는 RadixAttention 캐시 히트로 추가 10~20% 이득이 붙는다.

**언제 선택하는가**: RAG, 에이전틱 워크플로우처럼 공유 프리픽스가 많은 워크로드이거나, 구조화 출력이 핵심인 서비스이거나, DeepSeek 계열 모델을 서빙할 때.

---

### TensorRT-LLM — NVIDIA 하드웨어의 최대치를 끌어내는 옵션

NVIDIA가 만든 프레임워크로, NVIDIA GPU 전용이다<Ref id={10} />. TensorRT 엔진 컴파일 기반으로 동작하기 때문에 배포 전 모델 컴파일 단계가 필수다.

FP8, FP4, INT4, NVFP4 등 저정밀도 최적화에서 타의 추종을 불허한다. H100의 Transformer Engine을 풀로 활용하는 FP8 연산, Blackwell(B200)의 FP4 지원이 포함된다. p95 TTFT 기준으로 가장 낮은 지연을 기록한다<Ref id={11} />.

NVIDIA Triton Inference Server와 결합해 프로덕션 배포 레이어를 구성하는 것이 표준 패턴이다. 2025년에 OpenAI 호환 프론트엔드가 안정(stable) 상태로 전환됐다.

GitHub Stars는 13,000 수준<Ref id={12} />으로 vLLM/SGLang보다 적다. 초기 컴파일 시간, NVIDIA 전용이라는 제약, 높은 설정 복잡도가 진입 장벽이다.

**언제 선택하는가**: NVIDIA GPU를 사용하고, 하드웨어를 고정할 수 있으며, 설정 복잡도를 감수하고 최대 성능을 원할 때.

---

### H100 벤치마크 비교

아래 표는 H100 단일 GPU, Llama 3.1 8B 모델 기준 세 프레임워크를 비교한다. 조건이 다르면 수치가 달라지므로 같은 조건에서 직접 측정한 값을 참고 기준으로만 사용해야 한다<Ref id={9} /><Ref id={11} />.

| 지표 | vLLM | SGLang | TensorRT-LLM |
|------|------|--------|--------------|
| 처리량 (tok/s, H100, Llama 3.1 8B) | ~12,500 | ~16,200 | 최고 (고병렬) |
| p95 TTFT (100 요청 동시) | ~1,450ms | 중간 | ~1,280ms |
| 설정 복잡도 | 낮음 | 낮음 | 높음 (컴파일) |
| 하드웨어 범용성 | 매우 높음 | 높음 | NVIDIA 전용 |
| 커뮤니티 규모 | ★★★★★ | ★★★★ | ★★★ |

H100 8× 클러스터, Llama 4 70B FP8, 배치 128 기준으로는 TensorRT-LLM 4,800 tok/s, vLLM 3,400 tok/s, TGI 2,900 tok/s 순서다<Ref id={13} />.

SGLang의 처리량 우위는 RadixAttention의 캐시 재사용 효과다. 프리픽스 공유율이 낮은 워크로드에서는 vLLM과 차이가 줄어든다.

## TGI의 퇴장 서사

Text Generation Inference(TGI)는 Hugging Face가 만든 프레임워크로, 한때 HF 생태계의 기본 서빙 레이어였다.

2024년 12월 TGI v3.0을 출시하며 공격적인 성능 수치를 발표했다. 단일 NVIDIA L4 GPU(24GB)에서 Llama 3.1-8B 기준 vLLM 대비 3배 처리량, 긴 프롬프트에서 vLLM 대비 13배 빠름이라는 결과였다<Ref id={14} />.

1년 뒤인 2025년 12월 11일, TGI는 유지보수 모드(Maintenance Mode)로 전환됐다<Ref id={15} />. 버그 픽스와 경량 문서 개선만 수락하고, 신규 기능 개발은 없다. HF 공식 README에서는 vLLM과 SGLang을 대체 프레임워크로 명시했다.

무슨 일이 있었던 걸까. TGI의 선택은 단일 추론 엔진 대신 멀티 백엔드 프론트엔드로의 포지셔닝 전환이었다. TensorRT-LLM, vLLM, llama.cpp, AWS Neuron, Google TPU를 백엔드로 연결하는 통합 프론트엔드가 되겠다는 방향이다. 그러나 vLLM과 SGLang이 자체적으로 빠르게 발전하면서 TGI가 프론트엔드 레이어로서 제공하는 부가가치가 얇아졌고, 커뮤니티 기여도 집중되지 않았다.

v3.0의 13배 수치는 조건이 특수했다 — 긴 프롬프트, 단일 L4 GPU. 범용 워크로드에서의 비교는 아니었다. 선택은 이미 커뮤니티의 손을 떠난 상태였다.

## 2군: 로컬/엣지 생태계

### llama.cpp — CPU에서 LLM을 실행한다는 것의 의미

Georgi Gerganov가 C/C++로 작성한 추론 엔진이다<Ref id={16} />. 2023년 처음 공개될 때 "MacBook에서 LLaMA를 실행한다"는 주제 하나로 HN 프론트 페이지를 장악했다. 그 이후 3년간 LLM 로컬 실행의 기반이 됐다.

2026년 3월 GitHub Stars 100,000을 달성했다. 3년 이내에 달성한 최초의 오픈소스 AI 프로젝트라는 기록이다<Ref id={16} />. 2026년 2월에는 ggml-org 팀 전체가 Hugging Face에 합류했다.

핵심은 **GGUF 포맷**이다. 모델 가중치, 토크나이저, 메타데이터를 단일 파일로 번들하고, Q2~Q8까지 다양한 비트폭 양자화를 지원한다. Q4\_K\_M은 FP16 대비 크기 75% 감소, 품질 92% 유지가 가능한 권장 스위트 스팟이다.

CPU/GPU 혼합 추론이 특기다. 모델 레이어 일부를 CPU RAM에, 나머지를 VRAM에 올리는 방식으로 VRAM이 부족한 환경에서도 실행이 가능하다. Apple Silicon의 통합 메모리 아키텍처와 특히 잘 맞는다.

llama-server를 통해 OpenAI 호환 REST API를 제공하고, 2025년에는 멀티 모델 동적 로드/언로드를 지원하는 Router Mode가 추가됐다.

**Ollama, LM Studio, GPT4All, KoboldCpp는 모두 llama.cpp를 추론 엔진으로 사용한다.** llama.cpp는 이 생태계 전체의 기반이다.

---

### Ollama — Docker처럼 모델을 다루는 UX

llama.cpp 기반의 래퍼 도구다. Docker의 이미지 관리 UX에서 영감을 받아 LLM을 컨테이너처럼 관리한다<Ref id={17} />.

```bash
ollama pull llama3.2
ollama run deepseek-r1
```

2026년 초 기준 GitHub Stars 110,000+, 월간 다운로드 52M(2026 Q1)<Ref id={17} />. CES 2026에서 NVIDIA 최적화를 통해 토큰 생성 속도를 최대 35% 향상시켰다.

OpenAI 호환 HTTP API를 내장하고 있어 기존 OpenAI SDK를 쓰는 코드의 `base_url`만 바꿔도 연동된다.

GUI 오버헤드가 없어 LM Studio 대비 20~30% 빠른 토큰 생성이 가능하다는 실측이 있다. 실무에서는 "LM Studio로 모델 평가 → Ollama로 개발 → vLLM으로 프로덕션 배포"의 3단계 워크플로우가 자주 쓰인다.

---

### LiteLLM — 100개 이상의 LLM API를 하나로

BerriAI가 만든 게이트웨이 도구다<Ref id={18} />. OpenAI, Anthropic, Vertex AI, Bedrock, Azure, Cohere 등 100개 이상의 LLM을 단일 OpenAI 호환 인터페이스로 통합한다.

추론 엔진이 아니다. 프록시 서버로 배포해 가상 키 관리, 팀별 예산/속도 제한, 비용 추적, PII 마스킹, 콘텐츠 필터링을 중앙에서 처리한다. MCP/에이전트 게이트웨이로도 사용된다.

멀티 프로바이더 환경에서 공통 추상화 레이어가 필요할 때, 또는 셀프호스팅 인프라와 클라우드 API를 함께 쓸 때 유용하다.

## 3군: 특수 목적 프레임워크

아래 네 프레임워크는 특정 맥락에서 의미 있는 선택지다. 범용 서빙 스택의 대안보다는 특수 요구사항의 해결책으로 보는 게 맞다.

**LMDeploy (InternLM/OpenMMLab)**: Shanghai AI Laboratory가 InternLM 계열 모델을 위해 만든 프레임워크다<Ref id={19} />. TurboMind 엔진이 핵심으로, H800에서 vLLM 대비 1.5~1.8배 처리량을 기록한다. W4A16 양자화, FP8 MoE 최적화, Prefill/Decode 분리(Mooncake/DLSlime 연동)를 지원한다. GitHub Stars 7,600 수준이며, 중국 연구 기관과 기업 환경에서 주로 쓰인다.

**DeepSpeed-MII (Microsoft)**: Microsoft의 DeepSpeed 프레임워크 위에서 동작하는 서빙 레이어다<Ref id={20} />. Dynamic SplitFuse 기법으로 고정 길이 청크를 통한 지속적 배치 처리를 구현해 vLLM 대비 최대 2.5배 처리량을 주장한다. 그러나 2025년 이후 의미 있는 신규 기능 업데이트가 확인되지 않으며, vLLM/SGLang 대비 커뮤니티 활성도가 낮다. 주로 Microsoft Azure 생태계 내부에서 활용된다.

**MLC LLM (MLC AI)**: Apache TVM 컴파일러 기반으로 대상 하드웨어에 맞춰 커널을 자동 최적화하는 프레임워크다<Ref id={21} />. 브라우저 내 WebGPU 기반 추론(WebLLM), iOS, Android, Apple Metal 지원이 특기다. ICLR 2025에 논문이 게재됐으며, 모바일/엣지/브라우저 배포가 목표인 경우 실질적인 선택지다.

**Modular MAX (Modular)**: Mojo 언어 기반으로 CUDA 없이 고성능 GPU 커널을 작성하는 접근을 취한다<Ref id={22} />. NVIDIA, AMD, Apple Silicon, Intel, ARM을 단일 컨테이너로 지원하며 vLLM 대비 2배 성능을 주장한다. 2025년 7월 MAX Python API 전체를 오픈소스화했고, 2026년 1월 정식 출시됐다. 아직 신생 단계지만 CUDA 의존성을 줄이려는 시도에서 주목할 만하다.

## 선택 기준 요약

상황을 먼저 정의하면 선택은 좁아진다.

**GPU 서버, 빠른 시작이 목표**: vLLM. 설정이 가장 간단하고, 하드웨어 선택의 여지가 생긴다.

**RAG, 에이전트, 구조화 출력이 핵심**: SGLang. RadixAttention의 프리픽스 캐시 재사용이 직접적인 이득을 준다.

**NVIDIA GPU, 최고 성능이 요건**: TensorRT-LLM. 설정 복잡도를 감수할 팀 역량이 있을 때.

**로컬 개발, GPU 없는 환경**: llama.cpp + Ollama. 두 도구를 함께 쓰는 게 일반적이다.

**여러 LLM 프로바이더를 동시에 사용**: LiteLLM.

**모바일/브라우저 배포**: MLC LLM.

---

2편에서는 "어떤 프레임워크가 있는가"를 다뤘다. 3편은 그 다음 단계다 — 프레임워크를 골랐다면, 어디에 어떻게 배포할 것인가. 클라우드 플랫폼 선택, Kubernetes 오케스트레이션, GPU 비용 최적화, 그리고 TTFT/처리량 모니터링까지. 프레임워크가 돌아가는 환경 전체를 다룬다.
