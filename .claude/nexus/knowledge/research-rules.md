# Research Rules

<!-- tags: research, source, verification, report, posting, image -->

연구팀([research]/[research!])이 조사 및 포스트 작성 시 따르는 규칙.

---

## 1. 출처/검증

### 출처 등급 (티어 시스템)

| 티어 | 출처 유형 | 사용 규칙 |
|------|----------|-----------|
| T1 | peer-reviewed 논문, 공식 기술 보고서 | 제한 없이 인용 |
| T2 | 공식 블로그, 컨퍼런스 발표, 시스템 카드 | 인용 가능, 출처 명시 |
| T3 | 기술 미디어, 개인 블로그, 커뮤니티 정리 | 교차 검증 필요 (T1/T2와 대조 후 사용) |

### 비공개 모델 정보

공식 발표 + 근거 있는 추론 허용. 공식 미확인 정보는 불확실성을 명시적으로 표기:
- "~로 알려져 있다", "~로 추정된다" 등 톤으로 구분
- 다수의 독립적 출처가 일관되게 지지하는 추론만 허용

### 수치 검증

- **핵심 주장** (성능 비교, 비용, "최초" 클레임): 1차 출처 대조 필수
- **부수적 수치** (파라미터 수, 연도 등): 다수 출처 일치로 충분
- **T3 출처의 핵심 수치**: 반드시 교차 검증

### 포스트 본문 표기

- **인라인 톤 차등**: T1/T2는 확신 있는 서술, T3는 출처에 따라 확신도를 다르게
- **하단 출처 목록**: 각 출처에 티어 태그 표기 (`[T1]`, `[T2]`, `[T3]`)
- **시각적 callout**: 기본 사용 안 함. 특별한 경우에만 선택적 사용

## 2. 리포트 양식

### 필수 구조

```markdown
# {편} 조사 리포트: {제목}

조사일: {YYYY-MM-DD}

---
## 조사 결과
### {주제별 섹션}
(출처/근거 인라인 포함)
```

### 선택 확장 (해당 시에만 추가)

- **보완 내역**: 보완이 발생하면 반드시 기록 (보완일, 사유, 변경 항목)
- **검증 상태**: postdoc 검증 결과 기록

### 네이밍

- 시리즈: `{시리즈 prefix}-{편번호}-research.md`
- 단독: `{주제}-research.md`
- 저장 위치: `.nexus/reports/`

## 3. 포스트 변환

### 톤/깊이

- **타겟**: 실무자 — "이전 방식보다 왜 나은지" 직관적으로 설명 가능한 수준
- **전문 용어**: 사용하되 첫 등장 시 설명
- **수식**: 핵심 1-2개 포함, 직관적 해설 동반
- **톤**: 친근하지만 명료하게

### 시리즈 구성

같은 시리즈 조건: **주제 범위 + 톤/깊이** 모두 일관해야 함. 하나라도 달라지면 새 시리즈로 분리.

새 시리즈 분리 시: 기존 시리즈와 내용적 연결이 있으면 이전 시리즈의 마지막 편을 살펴보고 브릿지가 자연스러운지 확인. 필요시 마지막 편에 연결 문구 추가.

### 분량

한 편이 **하나의 완결된 개념**을 독립적으로 설명하는 것이 기준. 분량 제한 없이, 주제 복잡도에 따라 자연스럽게 결정.

## 4. 이미지

### 제작 정책

- **자체 제작 기본** — 논문 figure 직접 사용 안 함
- **우선순위**: SVG > Mermaid > 없음
- 에이전트가 SVG/Mermaid로 자동 생성, 사용자가 검토 후 필요시 수동 교체/보강
- 참고한 원본이 있으면 "~를 참고하여 재구성" 표기

### 사용 기준

시각화가 이해에 기여할 때 권장 (필수 아님). 참고 매핑:

| 주제 유형 | 권장 이미지 |
|-----------|------------|
| 아키텍처 설명 | 블록 다이어그램 |
| 알고리즘/프로세스 | 플로우차트 |
| 기술 비교 | 비교 도식 |
| 시대별 흐름 | 타임라인 |
| 수식 직관화 | 개념 도식 |

### 저장

- 시리즈: `public/images/{시리즈 prefix}/`
- 단독: `public/images/{slug}/`
- MDX 참조: `/images/{폴더}/{파일명}`
