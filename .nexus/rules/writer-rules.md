# Writer Rules

라이터가 리서처의 조사 리포트를 블로그 포스트로 변환할 때 따르는 규칙.

---

## 1. 톤/깊이

- **타겟**: 실무자 — "이전 방식보다 왜 나은지" 직관적으로 설명 가능한 수준
- **전문 용어**: 사용하되 첫 등장 시 설명
- **수식**: 핵심 1-2개 포함, 직관적 해설 동반
- **톤**: 친근하지만 명료하게

---

## 2. 출처 표기

포닥이 리포트에 부여한 티어 태그를 기반으로 표기한다.

- **인라인 톤 차등**: T1/T2는 확신 있는 서술, T3는 출처에 따라 확신도를 다르게
- **하단 출처 목록**: 각 출처에 티어 태그 표기 (`[T1]`, `[T2]`, `[T3]`)
- **시각적 callout**: 기본 사용 안 함. 특별한 경우에만 선택적 사용

---

## 3. 시리즈 구성

같은 시리즈 조건: **주제 범위 + 톤/깊이** 모두 일관해야 함. 하나라도 달라지면 새 시리즈로 분리.

새 시리즈 분리 시: 기존 시리즈와 내용적 연결이 있으면 이전 시리즈의 마지막 편을 살펴보고 브릿지가 자연스러운지 확인. 필요시 마지막 편에 연결 문구 추가.

### 포스트 간 연결 문구

시리즈 내 포스트는 다음 패턴을 따른다:

**도입부** (첫 단락 근처):
- 이전 편 핵심 1~2문장 요약 → 이번 편 주제로 전환
- 예: "5편에서 오픈소스 생태계의 폭발적 성장을 살펴봤다. ... 6편은 그 질문에 대한 답들을 해부한다."

**마무리** (본문 끝):
- 다음 편 주제 예고 (제목/핵심 질문 힌트)
- **시리즈 확장 가능성을 닫지 않는다** — "마지막이다", "이것으로 끝이다" 같은 단정적 표현 금지
- 예: "다음 편(8편)은 전환점이다. 생각에서 **행동**으로 — LLM이 도구를 호출하고..."

**시리즈 최종편만** "시리즈를 마치며" 섹션 허용. 단, 최종편 여부는 리드가 확정.

---

## 4. 분량

한 편이 **하나의 완결된 개념**을 독립적으로 설명하는 것이 기준. 분량 제한 없이, 주제 복잡도에 따라 자연스럽게 결정.

---

## 5. 이미지

### 제작 정책

- **자체 제작 기본** — 논문 figure 직접 사용 안 함
- **우선순위**: SVG > Mermaid > 없음
- 라이터가 SVG를 직접 생성한다 (스타일 가이드 준수)
- 참고한 원본이 있으면 "~를 참고하여 재구성" 표기

### SVG 스타일 가이드

시리즈 내 스타일 일관성을 유지한다. 새 시리즈 시작 시 첫 SVG가 기준이 된다.

공통 원칙:
- 배경: 투명 (`fill="none"`)
- 선/텍스트: 단일 색상 계열 유지 (예: `#374151`)
- 박스: 둥근 모서리 (`rx`), 파스텔 계열 fill
- 화살표: `<marker>` 또는 `marker-end` 정의
- 다크 모드 호환: 배경 투명 + 텍스트 색상은 밝은/어두운 배경 모두에서 보이는 중간톤

기존 시리즈 참고:
- `llm-anatomy`: monospace 폰트, `#374151`, 파스텔 fill (EDE9FE, D1FAE5, FEF3C7 등)
- `llm-eng`: system-ui 폰트, `#1E40AF` 타이틀, 더 다양한 컬러

### alt 텍스트

**도식 내용을 텍스트로 풀어 설명한다.** 단순 제목이 아닌 다이어그램이 전달하는 정보를 서술형으로 기술.

```
✗ alt="MCP 아키텍처"
✓ alt="MCP 아키텍처 — Host, Client, 여러 MCP Server가 JSON-RPC 2.0으로 통신하는 M+N 통합 구조"
```

### 사용 기준

아래 주제 유형에 해당하는 포스트는 **최소 1개 SVG 필수**. 해당하지 않는 단순 글(짧은 뉴스, 공지 등)은 면제.

| 주제 유형 | 필수 이미지 |
|-----------|------------|
| 아키텍처 설명 | 블록 다이어그램 |
| 알고리즘/프로세스 | 플로우차트 |
| 기술 비교 | 비교 도식 |
| 시대별 흐름 | 타임라인 |
| 수식 직관화 | 개념 도식 |

### 셀프 체크

작성 완료 전 반드시 확인:

- [ ] 위 주제 유형 매핑에 해당하는가? → 해당하면 SVG가 포함되어 있는지 확인
- [ ] SVG 스타일 가이드(색상, 배경, 다크모드 호환) 준수 여부
- [ ] alt 텍스트가 도식 내용을 서술형으로 설명하는지 확인

### 저장

- 시리즈: `public/images/{시리즈 prefix}/`
- 단독: `public/images/{slug}/`
- MDX 참조: `/images/{폴더}/{파일명}`

## 6. MDX 주의사항

### 한글 bold 패턴

`**텍스트)**조사` 패턴(bold 닫기 직후 한글 조사)은 렌더링 오류를 유발한다. `<strong>` 태그를 직접 사용하거나 bold와 조사 사이에 공백을 둔다.

```
✗ **Attention)**의 복잡도
✓ <strong>Attention)</strong>의 복잡도
```

빌드 시 `fix-mdx-bold.mjs`가 자동 교정하지만, 처음부터 피하는 것이 원칙이다.

### 프론트매터

`series`, `seriesOrder`, `category`, `slug`, `date`, `published`는 리드가 정의한 초안을 따른다. 라이터는 `title`, `description`(max 200자), `tags`를 본문에 맞게 작성/조정한다.

`category`는 코드에 정의된 값만 사용: **Web, AI, MLOps, DevOps, General**
